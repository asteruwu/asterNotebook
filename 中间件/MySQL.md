# 基础架构

## 体系架构分层设计

- **客户端连接层 (Connection Pool)**
  - **核心职责**：管理客户端连接、授权认证与安全防御。
  - **组件机制**：通过连接池复用线程，阻断频繁创建/销毁的资源开销；建连时执行账号鉴权并拉取权限树缓存于会话句柄。

- **核心服务层 (Server Layer)**
  - **核心定位**：跨存储引擎的核心逻辑层，所有 SQL 的解析、优化与调度均在此完成。
  - **SQL 解析器 (Parser)**：词法+语法分析，将 SQL 字符流构建为**抽象语法树 (AST)**。
  - **查询优化器 (Optimizer)**：
    - 基于启发式规则 (Rule-Based) 与代价模型 (Cost-Based) 敲定执行策略。
    - 执行谓词下推、Join Reordering，选择最优索引，输出**执行计划 (Execution Plan)**。
  - **执行器 (Executor)**：
    - 进行表级与字段级运行时权限校验。
    - 调用底层存储引擎的 **Handler API**，循环拉取满足谓词条件的数据行并返回。

- **存储引擎层 (Engine Layer)**
  - **核心定位**：负责数据的物理落地与读取，向 Server 层暴露统一 Handler API。
  - **架构设计特色**：**插件式 (Pluggable) 架构**。Server 层与物理存储彻底解耦，InnoDB、MyISAM 等引擎对 Server 层呈现高度一致的接口。

- **文件系统层**
  - **物理落地**：依赖 OS 文件系统实现持久化存储。
  - **核心组件**：数据文件（`.ibd` 等）与三大核心日志（Redo Log、Undo Log、Binlog）。

## 一条 SQL 的完整执行链路

### DQL (数据查询语言) 执行生命周期

> [!NOTE]
>
> **【核心链路】跨层级组件协同与查询流转机制**

1. **连接器 (Connector)**：
   - 负责 TCP 建连与用户身份鉴权，完成 Session 上下文初始化。
   - 连接时拉取全局权限树并缓存于当前连接句柄中。
2. **查询缓存 (Query Cache)**：*（MySQL 8.0 彻底废弃）*
   - 以 K-V 结构缓存 `SELECT` 结果集。
   - 写操作即触发全量缓存失效，所引发的锁粒度竞争是其被淘汰的根因。
3. **解析器 (Parser)**：
   - **词法分析 (Lexical Analysis)**：基于有穷状态机拆解 SQL 字符流为标识符 (Token)。
   - **语法分析 (Syntactic Analysis)**：依据 SQL 语法规则，构建并校验**抽象语法树 (AST)**。
4. **优化器 (Optimizer)**：
   - 基于启发式规则 (Rule-Based) 与代价模型 (Cost-Based) 敲定最优执行策略。
   - 执行谓词下推、表连接顺序枚举 (Join Reordering)。
   - 在聚簇索引与二级索引中计算最小 IO/CPU 成本，输出执行计划 (`Execution Plan`)。
5. **执行器 (Executor)**：
   - 进行表级别与字段级别的 DML 运行时权限校验。
   - 调用底层存储引擎的 **Handler API**，循环拉取满足谓词条件的数据行并聚合返回。

### DML (数据操纵语言) 与事务的两阶段提交

> [!IMPORTANT]
>
> **【重难点/架构】崩溃恢复与跨节点状态机的强一致保障**

DML 操作（如 `UPDATE`）的核心异构在于**日志子系统的多阶段协同**。InnoDB 面向"单机 Crash-Safe"与"集群 Replication"两大场景，实现了基于 Redo Log 与 Binlog 的**两阶段提交协议 (Two-Phase Commit, 2PC)**。

- **阶段零：内存寻址与版本切分**
  - **Buffer Pool 命中**：执行器向引擎层下发更新指令，InnoDB 检查目标页是否驻留于缓冲池；未命中则触发物理磁盘 IO 页加载。
  - **Undo Log 分配**：引擎提取旧行镜像落盘至 Undo Log 段，维系 Rollback 能力并构建 MVCC 版本链。
  - **脏页构造 (Dirty Page)**：引擎对 Buffer Pool 中的内存数据执行覆写，内存态超前于磁盘态。

- **阶段一：Prepare 预提交 (Engine 层)**
  - **物理重做录入**：InnoDB 将物理增量写入 Redo Log Buffer，依 `innodb_flush_log_at_trx_commit` 策略发起 `fsync` 刷盘。
  - **状态封存**：事务打上 `Prepare` 状态标记。

- **中间态：Binlog 归档 (Server 层)**
  - **逻辑日志落地**：执行器生成变更语义的 Binlog，按 `sync_binlog` 阈值完成物理刷盘。
  - **为何 Binlog 归属 Server 层**：
    - Binlog 是跨所有存储引擎共享的全局逻辑归档与主从同步机制，与引擎实现无关。
    - 若下沉至引擎层，无法维系跨引擎间一致的逻辑复制语义结构。

- **阶段二：Commit 决议与交割 (Engine 层)**
  - InnoDB 将 Redo Log 状态由 `Prepare` 冲正为 `Commit`。
  - 释放该事务占据的行锁 (Record Lock)，事务终止。

### 一致性定界与崩溃恢复机理

> **分布式状态机对齐：何为一致性？**
- 特指 **Engine 层的物理状态推演 (Redo Log) 与 Server 层的拓扑复制总线 (Binlog) 之间的状态强一致**。
- 若割裂：主库凭 Redo 还原本地数据，从库因 Binlog 断裂而未同步，导致集群**主备数据脑裂 (Split-Brain)**。

> **Crash Recovery: 两阶段防线下的容错流转**

MySQL 重启后拉起恢复线程，基于事务的全局唯一 ID (XID) 扫检日志：
1. **Redo 处于 Prepare 态，Binlog 缺失或损坏**：                                                                                                                                          
   - 从节点未接收到该事务的 Binlog，主从尚未同步。
   - **决议**：执行 **Rollback**。引擎寻址 Undo Log 段，逆向重演变动前版本，覆盖脏数据。
2. **Redo 处于 Prepare 态，Binlog 完整留存**：
   - 逻辑归档完毕，从库已接收增量，仅引擎层缺失 Commit 状态提交。
   - **决议**：执行**自动 Commit**，引擎将 Prepare 强行冲正为 Commit，保全集群数据一致。

# 存储

## InnoDB vs MyISAM 存储引擎

> [!NOTE]
> **【核心定位】事务安全型引擎与读操作优化型引擎的底层构型分野**

- **并发控制与锁粒度**
  - **InnoDB**：
    - 默认行级锁 (Row-Level Lock) + **MVCC**，读写解耦互不阻塞，承载超高并发 OLTP。
    - 无索引命中时退化为表锁。
  - **MyISAM**：
    - 仅支持表级锁 (Table-Level Lock)，写 DML 排他锁定全表。
    - 并发写吞吐极低，高并发下线程资源极易饥饿雪崩。

- **事务支持**
  - **InnoDB**：完整支持 ACID 事务。Undo Log 保障原子性与回滚能力；Redo Log + WAL 保障持久性。
  - **MyISAM**：无事务边界与回滚支持，DML 执行即即时固化。

- **崩溃恢复 (Crash Safety)**
  - **InnoDB**：具备完备的 Crash-Safe 自愈能力。
    - 重启时恢复线程扫描 Redo Log，将数据页补齐至宕机前内存态，此动作称为**前滚 (Roll-Forward)**。
    - 随后基于 Undo Log 对未 Commit 的残损事务执行逆向覆盖清除，即**回滚 (Rollback)**。
  - **MyISAM**：不具备 WAL 日志保护体系。
    - B+ 树节点分裂合并期间若遭遇断电，`.MYI` 索引文件极大概率产生结构性断层损坏。
    - 恢复须借助离线工具 `myisamchk` 全表重建索引。

- **底层物理存储模型**
  - **InnoDB**：**聚簇索引 (Clustered Index) 架构**。
    - 数据行与主键 B+ 树捆绑存储于独立表空间文件 (`.ibd`)。
    - 主键索引叶子节点直接承载完整行数据。
  - **MyISAM**：**非聚簇索引 (Non-Clustered Index) 架构**，数据与索引在物理文件层级彻底剥离。
    - `.MYD`（MyISAM Data）：真实数据行序列化落盘文件。
    - `.MYI`（MyISAM Index）：全部 B+ 树索引结构文件，索引叶子节点仅存储指向 `.MYD` 对应行偏移量的物理指针。

- **外键约束**
  - **InnoDB**：引擎层原生支持跨表参照完整性。*(注：高并发微服务架构中，DB 层外键通常引发锁竞争瓶颈，业界规范倾向上移至应用层校验)*
  - **MyISAM**：不支持。

- **场景选型**
  - MySQL 5.5.5 起 InnoDB 确立为默认引擎，统领绝大多数 OLTP 核心链路。
  - MyISAM 仅残存于极少数无事务诉求、读极多写极少的边缘历史包袱场景。

## B+ 树索引存储架构

> [!NOTE]
> **【核心定位】磁盘友好的平衡多路搜索树，InnoDB 一切索引的基石**

![MySQL-B+树示意图.drawio](C:\Users\86133\Desktop\学习\dio图表\SRE\MySQL-B+树示意图.drawio.svg)

### 聚簇索引

- **存储形态**：
  - InnoDB 必须且只有一棵聚簇索引树。
  - 叶子节点 (Leaf Node) 直接承载完整行数据；非叶子节点仅存储索引键值与子节点指针。
- **主键选取优先级**：显式 PRIMARY KEY → 第一个非空 UNIQUE 索引 → 引擎自动生成的隐式 6 字节 RowID。
- **物理有序性**：数据页按主键升序排列存储于磁盘，相邻主键数据物理连续，区间扫描 IO 代价极低。
- **设计推论**：主键建议使用自增整型。
  - 随机 UUID 导致每次插入在 B+ 树中间页随机寻址，引发**页分裂 (Page Split)** 与大量随机 IO。

### 二级索引与回表

- **存储形态**：二级索引叶子节点**不存储完整行数据**，仅存储**索引列值 + 主键值 (PK)**。
- **回表查询 (Table Lookup)**：
  - 通过二级索引定位主键值后，须再次以主键为键在聚簇索引树执行完整检索，取回整行数据。
  - 此二次查询即为回表，引发额外 IO 开销。
- **覆盖索引 (Covering Index)**：
  - 若 `SELECT` 所需全部字段均已包含于二级索引叶子节点（索引列 + PK），引擎直接从索引树返回，无须回表。
  - `EXPLAIN` 中 `Extra: Using index` 即表征覆盖索引命中。

> 为什么不用其他数据结构？

| 淘汰候选 | 致命缺陷 |
|---|---|
| **哈希索引 (Hash)** | 仅支持等值查询，无法支持范围查询 (`BETWEEN`/`>`/`<`) 与排序；哈希冲突下退化为链表；无法利用索引完成 `ORDER BY`。 |
| **二叉搜索树 (BST)** | 数据有序插入时退化为链表（$O(n)$ 查询），磁盘 IO 次数等于树深，海量数据下深度极深。 |
| **红黑树 (Red-Black Tree)** | 自平衡解决了退化问题，但仍是二叉结构，树深 $O(\log_2 n)$，百万级数据深度高达 20+ 层，每层一次磁盘 IO，性能灾难。 |
| **标准 B 树 (B-Tree)** | 数据行同时存于内部节点与叶子节点，内部节点因承载数据而膨胀，每个磁盘页能容纳的索引键数量大幅缩减，树高增加；范围查询无法利用叶子节点链表顺序遍历，须回溯父节点。 |

- **B+ 树的核心优势**：
  - **矮胖形态**：所有数据下压至叶子层，内部节点只存键值与指针，单页承载更多索引键，树高被压到 3~4 层。
  - **叶子节点双向链表**：相邻叶子节点通过指针串联，范围查询找到起点后沿链表顺序扫描，IO 极度顺序化。
  - **磁盘页对齐**：节点大小恰好对齐 OS 页（默认 16KB），单次 IO 一次性载入整个节点，充分压榨磁盘预读 (Read-Ahead) 特性。



# 索引

## 索引高级特性与失效防雷

### 最左前缀匹配原则

- **核心机制**：
  - 联合索引 `(a, b, c)` 的 B+ 树按 a → b → c 的优先级多重排序。
  - 优化器只能从最左列起连续命中，遇到断层或范围谓词后，该列之后的所有列退出索引检索路径。
- **典型场景推演**（索引 `(last_name, first_name, age)`）：
  - `WHERE last_name='张'`：命中第 1 列，索引有效。
  - `WHERE last_name='张' AND first_name='三'`：命中前 2 列，索引有效。
  - `WHERE last_name='张' AND age=18`：`last_name` 命中，`first_name` 断层，`age` 退出索引检索路径，降级为叶子节点扫描过滤。
  - `WHERE first_name='三'`：跳过最左列 `last_name`，索引完全失效，全表扫描。
  - `WHERE last_name='张' AND first_name LIKE '三%' AND age=18`：前缀匹配 `first_name` 可命中，但其为范围查询，`age` 列索引退出。

> [!NOTE]
>
> **【核心本质】数据存在 ≠ 树能定位**
> - `age` 值物理上存在于叶子节点，但非叶节点中不存在跨 `first_name` 分组的 `age` 排序路径。
> - 引擎无法对 `age` 做二分定位 seek，只能全量扫描命中范围内的叶子节点逐行比对。
> - **索引失效 = 寻址路径缺失；ICP = 利用叶子节点已有数据就地过滤，省去无效回表。**

### 索引下推

> [!NOTE]
> **【优化器特性】将 WHERE 过滤语义从 Server 层下推至 Engine 层**

- **无 ICP 时的路径**：
  - Engine 层按联合索引命中前缀，定位一批叶子节点并取出所有 PK。
  - 全量回表取完整行，再由 Server 层执行 WHERE 后缀字段过滤，大量无效回表 IO 浪费。
- **ICP 开启后的路径**：
  - Engine 层在联合索引叶子节点中，对后缀字段（即使无法参与 B+ 树 seek 定位）**提前执行条件过滤**。
  - 仅将满足所有索引列条件的行回表，大幅削减无效 IO。
- **感知方式**：`EXPLAIN` 输出 `Extra: Using index condition` 即表征 ICP 生效。

### 常见索引失效场景

| 失效场景 | 根因 |
|---|---|
| **函数包裹索引列** `YEAR(create_time) = 2024` | B+ 树存储原始值，套函数后键值语义破坏，优化器无法确定范围起点 |
| **表达式运算** `id + 1 = 10` | 同上 |
| **隐式类型转换** `phone VARCHAR` 列执行 `WHERE phone = 13812345678` | MySQL 对字段列执行隐式 CAST，等价于对索引列施加函数 |
| **左模糊匹配** `WHERE name LIKE '%张'` | B+ 树按字符串前缀排序，前缀未知，无法定位检索起点 |
| **OR 扩散失效** `WHERE indexed_col=1 OR non_indexed_col=2` | OR 连接列若非均有索引，优化器退化为全表扫描 |
| **联合索引非最左** `WHERE b=1 AND c=2`（跳过 `a`） | 未从最左列起连续命中，寻址路径缺失 |

## 慢查询优化

### 执行计划深度研读

`EXPLAIN` 输出优化器为当前 SQL 敲定的执行计划，是慢查询诊断的核心工具。重点关注以下字段：

- **`type`（访问方法，性能递减）**：

  | 类型 | 含义 |
  |---|---|
  | `const` / `eq_ref` | 主键或唯一索引等值命中，至多一行，最优 |
  | `ref` | 非唯一索引等值查找，返回少量行 |
  | `range` | 索引范围扫描（`BETWEEN`/`>`/`<`/`IN`） |
  | `index` | 全索引扫描（遍历整棵索引树，仍优于全表） |
  | `ALL` | 全表扫描，性能最差，需立即排查 |

- **`key`**：实际命中的索引名，为 `NULL` 说明未使用任何索引。
- **`rows`**：优化器估算需要扫描的行数，数量越大代价越高。
- **`Extra`（关键劣化信号）**：

  | 值 | 含义 |
  |---|---|
  | `Using index` | 覆盖索引命中，无需回表，最优 |
  | `Using index condition` | ICP 生效，引擎层提前过滤 |
  | `Using where` | Server 层执行了额外过滤（索引无法完全覆盖 WHERE） |
  | `Using filesort` | 排序字段未走索引，需额外排序，性能告警 |
  | `Using temporary` | 使用了临时表（常见于 `GROUP BY`/`ORDER BY` 无法走索引时），严重告警 |

### 深分页破局

- **痛点根因**：
  - `LIMIT offset, n` 要求 MySQL 顺序扫描并丢弃前 `offset` 行，即便走索引也难逃大量宽列数据读取。
  - 时间复杂度 $O(offset)$，百万级偏移量时性能灾难。

- **方案一：延迟关联 (Deferred Join)**
  - 子查询仅扫描窄列 `id`，外层 JOIN 只对最终 10 行回表取全列，大幅削减总数据吞吐量。
  - **收益原理精确定界**：
    - **`ORDER BY id`（主键排序）**：窄列扫描（只读 `id`）远比宽列扫描（读 `SELECT *`）内存与 IO 开销轻。
    - **`ORDER BY created_at`（二级索引排序）**：子查询可命中**覆盖索引**，二级索引叶子节点存储 `(created_at, id)`，无需触碰聚簇索引。

  ```mysql
  -- 优化前（每行都要读取完整宽列，并丢弃 900 万行）
  SELECT * FROM orders ORDER BY id LIMIT 9000000, 10;
  
  -- 优化后（子查询仅扫窄列 id，外层仅对 10 行回表）
  SELECT * FROM orders
  INNER JOIN (
      SELECT id FROM orders ORDER BY id LIMIT 9000000, 10
  ) AS tmp USING(id);
  ```

- **方案二：游标翻页 (Keyset Pagination)**
  - 记录上一页末尾的主键值，下次查询从该主键之后开始，完全规避 offset 扫描。
  - 时间复杂度降至 $O(n)$，无论翻到第几页性能恒定。
  - **缺陷**：无法跳页，只能顺序翻页，适合无限滚动/分批拉取场景。

  ```mysql
  -- 第一页
  SELECT * FROM orders ORDER BY id LIMIT 10;
  
  -- 第 N 页（last_id 为上一页最后一行的 id，由应用层持有并传参）
  SELECT * FROM orders WHERE id > last_id ORDER BY id LIMIT 10;
  ```

  ```mysql
  SET GLOBAL long_query_time = 1;  -- 超过 1s 的查询记录入日志
  ```

  3. 验证 `EXPLAIN` 输出质变（`type: ALL` → `ref`/`range`）。

## ACID 特性与底层支撑组件

> [!NOTE]
> **【核心定位】事务四大约束属性及其底层工程实现分野**

| 特性 | 底层支撑 | 核心机制 |
|---|---|---|
| **原子性 (Atomicity)** | Undo Log | 事务失败时依据旧行镜像执行逆向覆盖，保障操作不可部分完成 |
| **一致性 (Consistency)** | A + I + D 合力 | 数据库约束 + 其他三特性共同保障，确保始终从一合法状态流转至另一合法状态 |
| **隔离性 (Isolation)** | MVCC + 锁机制 | 多版本快照机制解决读写冲突；行锁/表锁解决写写冲突 |
| **持久性 (Durability)** | Redo Log + WAL | 提交前物理增量已刷入 Redo Log，宕机后前滚恢复确保已提交数据不丢失 |

### 原子性支撑：Undo Log

- **物理形态**：
  - InnoDB 为每个写操作生成的旧行镜像快照。
  - 持久化存储于 Undo Tablespace 中。
- **回滚机制**：
  - 事务异常或主动 ROLLBACK 时，引擎逆向读取 Undo Log 链。
  - 依次执行反向操作（INSERT → DELETE，UPDATE → 还原旧值）。
  - 将数据页恢复至事务开始前的状态。
- **兼用于 MVCC**：Undo Log 链构成多版本历史版本链，是 MVCC 可见性判断的数据底座。

### 持久性支撑：Redo Log + WAL

- **WAL (Write-Ahead Logging) 原则**：
  - 数据页修改在写入物理文件（`.ibd`）之前，物理增量必须先顺序追加写入 Redo Log。
- **崩溃恢复**：
  - 重启时 InnoDB 扫描 Redo Log，对已 Commit 未刷盘的变更执行**前滚 (Roll-Forward)**。
  - 对仍处于 Prepare 态的残损事务，配合 Undo Log 执行**回滚 (Rollback)**。
- **刷盘策略（`innodb_flush_log_at_trx_commit`）**：

  | 值 | 写入位置（内存归属） | fsync 时机 | MySQL 进程 Crash | OS 宕机/断电 |
  |---|---|---|---|---|
  | `0` | MySQL Redo Log Buffer（**用户态**进程堆内存） | 每秒一次 | $\times$ 丢失最多 1s | $\times$ 丢失最多 1s |
  | `1`（默认） | 直接 fsync 至物理磁盘 | 每次 Commit | $\surd$ 不丢 | $\surd$ 不丢 |
  | `2` | OS Page Cache（**内核态**文件系统缓冲区） | 每秒一次 | $\surd$ 不丢 | $\times$ 丢失最多 1s |

  - **`0` vs `2` 本质差异**：
    - `0`：数据驻留**用户态**进程堆内存，进程 Crash 后内存随进程消失。
    - `2`：`write()` 系统调用将数据写入**内核态** OS Page Cache；MySQL 进程崩溃后 OS 仍存活，Page Cache 内容最终由 OS 刷入磁盘。

## 隔离级别与并发读写乱象

### 三大并发乱象

> [!IMPORTANT]
> **【正确理解前提】乱象是多事务并发场景下隔离不足的产物，非 Bug，是隔离级别权衡的代价。**

- **脏读 (Dirty Read)**：
  - 事务 A 读取到事务 B **尚未提交**的中间变更状态。
  - 若事务 B 随后回滚，事务 A 读取的数据在逻辑上从未存在。

- **不可重复读 (Non-Repeatable Read)**：
  - 同一事务内对**同一行**先后两次读取结果不一致。
  - 根因：其他事务在两次读取之间提交了 UPDATE/DELETE，关注点为**行级数据被修改**。

- **幻读 (Phantom Read)**：
  - 同一事务内对**同一范围条件**先后两次查询结果集行数变化。
  - 根因：其他事务在两次查询之间提交了 INSERT，关注点为**集合级记录数变化**。

### 四大隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 底层实现 |
|---|---|---|---|---|
| **Read Uncommitted** | $\times$ | $\times$ | $\times$ | 无任何隔离保护 |
| **Read Committed** | $\surd$ | $\times$ | $\times$ | 每次读取重新生成 Read View |
| **Repeatable Read**（MySQL 默认） | $\surd$ | $\surd$ | 快照读规避，当前读需 Next-Key Lock | 事务内首次读时生成 Read View；当前读加 Gap Lock |
| **Serializable** | $\surd$ | $\surd$ | $\surd$ | 所有读转化为当前读，串行加锁 |

## MVCC 并发控制引擎原理

> [!IMPORTANT]
> **【重难点/架构】读写不冲突并发机制的底座**

![MySQL-版本链维护.drawio](C:\Users\86133\Desktop\学习\dio图表\SRE\MySQL-版本链维护.drawio.svg)

### 底层建筑 | 隐藏字段 + Undo Log 版本链

InnoDB 为每行数据维护两个**隐式系统字段**，构建版本链的物理基础：

| 字段 | 含义 |
|---|---|
| `trx_id` | 最近一次修改该行的事务 ID |
| `roll_pointer` | 指向 Undo Log 中上一个历史版本的物理指针 |

- 每次 UPDATE/DELETE 时，引擎将旧行镜像写入 Undo Log 段。
- 通过 `roll_pointer` 将历次版本串成**单向链表（版本链）**，链头为最新版本，链尾为最旧版本。

### 核心组件 | Read View 与可见性四分法

Read View 是**轻量级并发状态元数据**，仅记录生成时刻活跃事务 ID 集合，不承载任何行数据。

- `m_ids`：生成时所有仍活跃（未提交）的事务 ID 集合。
- `min_trx_id`：`m_ids` 中的最小值。
- `max_trx_id`：当前系统已分配最大事务 ID + 1。
- `creator_trx_id`：当前事务自身 ID。

对版本链上每个版本的 `trx_id` 进行**四分法判定**：

| 条件 | 可见性 |
|---|---|
| `trx_id < min_trx_id` | Read View 生成前已提交 → **可见** |
| `trx_id >= max_trx_id` | Read View 生成后才开始 → **不可见** |
| `min_trx_id ≤ trx_id < max_trx_id` 且**在** `m_ids` 中 | 生成时仍未提交 → **不可见** |
| `min_trx_id ≤ trx_id < max_trx_id` 且**不在** `m_ids` 中 | 生成前已提交 → **可见** |

- 沿版本链从新到旧遍历，取第一个"可见"版本作为读取结果，全程无锁。

### 动静差异 | RC 与 RR 下 Read View 的生命周期

| 隔离级别 | Read View 创建时机 | 生命周期 | 效果 |
|---|---|---|---|
| **Read Committed** | 每次 `SELECT` 前重新创建（语句级） | 语句粒度，用完即弃 | 可见其他事务最新提交，产生**不可重复读** |
| **Repeatable Read** | 事务内首次 `SELECT` 时创建（事务级） | 事务粒度，整个事务复用同一个 | 始终读同一快照，**杜绝不可重复读** |

### 幻读破局 | MVCC + Next-Key Lock 联合拦截

- **快照读**（普通 `SELECT`）：
  - MVCC 读取 Read View 快照，后续 INSERT 的新行 `trx_id` 超出可见范围，天然不可见，**无幻读**。
- **当前读**（`SELECT FOR UPDATE` / `INSERT` / `UPDATE` / `DELETE`）：
  - 读取最新真实数据，新行可见，存在幻读风险。
  - InnoDB 通过 **Next-Key Lock（Record Lock + Gap Lock）** 锁定查询范围，阻止其他事务在该范围内插入新行。
- **残留边界**：
  - 同一事务内先快照读（未见幻行）后当前读（见到幻行），属幻读残留。
  - 生产中对关键范围查询应统一使用 `SELECT FOR UPDATE` 强制当前读加锁。

# 锁

## 锁的分类模型与宏观价值

> [!NOTE]
> **【核心定位】MVCC 解决读写冲突，锁解决写写冲突；二者分工协同共同支撑隔离性 (I)**

### 三维分类模型

**一、按粒度划分**

| 粒度 | 锁类型 | 适用场景 |
|---|---|---|
| 全局 | **全局锁 (Global Lock)** `FLUSH TABLES WITH READ LOCK` | 全库只读备份（备份期间阻塞所有写操作） |
| 表级 | **MDL 元数据锁 (Metadata Lock)** | DDL 与事务之间的表结构并发保护，Server 层自动加减 |
| 表级 | **表锁 (Table Lock)** `LOCK TABLES` | MyISAM 等无行锁引擎；InnoDB 极少使用 |
| 表级 | **意向锁 (Intention Lock)** IS / IX | 事务加行锁前先加意向锁，供表锁快速侦测行锁冲突 |
| 行级 | **行锁 (Row Lock)** | InnoDB 默认；并发写的最细粒度隔离单元 |

**二、按模式划分**

| 模式 | 符号 | 语义 | 兼容性 |
|---|---|---|---|
| **共享锁 (Shared Lock)** | S | 读锁，允许并发读 | S 与 S 兼容，S 与 X 不兼容 |
| **排它锁 (Exclusive Lock)** | X | 写锁，独占该行 | X 与任何锁不兼容 |

**三、按行锁类型划分（InnoDB 核心）**

| 类型 | 锁定范围 | 核心用途 |
|---|---|---|
| **记录锁 (Record Lock)** | 锁定单条索引记录 | 防止其他事务修改/删除该精确行 |
| **间隙锁 (Gap Lock)** | 锁定两条索引记录之间的间隙（不含端点） | 防止其他事务在间隙中插入新行，阻断幻读 |
| **临键锁 (Next-Key Lock)** | Record Lock + Gap Lock（左开右闭区间） | RR 级别下当前读的默认行锁形式，同时防修改与幻读 |
| **插入意向锁 (Insert Intention Lock)** | 特殊间隙锁，等待插入时占位 | 多事务同时插入同一间隙时互相不阻塞，提升并发 |

> 为何需要 Next-Key Lock 而非单独 Gap Lock？
- Gap Lock 仅防止区间内新行插入（"无中生有"），无法保护已有行被 UPDATE/DELETE。
- Record Lock 保护已有行的修改/删除，但不阻断新行插入。
- 范围查询的当前读需要两种保护同时在场，Next-Key Lock = Record Lock + Gap Lock 是二者的合体形态。

### 宏观架构价值

- **读写分工**：
  - MVCC 通过历史版本链实现**无锁快照读**，读不阻塞写，写不阻塞读。
  - 锁只作用于**写写冲突**（当前读场景），确保并发写的串行化安全。
- **粒度与并发的权衡**：
  - 全局锁 → 表锁 → 行锁，粒度递减，并发度递增，管理开销也递增。
  - InnoDB 以行锁为核心，通过意向锁向上传递加锁意图，避免表锁与行锁检测的全表扫描开销。
- **幻读防治**：
  - Gap Lock/Next-Key Lock 是 RR 级别下防幻读的物理屏障，锁住索引键值之间的区间，阻止该范围内新行插入。

## InnoDB 行锁算法

> [!IMPORTANT]
> **【重难点/架构】加锁范围由索引类型与查询类型共同决定**

### 加锁推演

**前置条件**：加锁仅发生在**当前读**（`SELECT FOR UPDATE` / `INSERT` / `UPDATE` / `DELETE`）；快照读（普通 `SELECT`）依赖 MVCC，不加锁。

**场景一：唯一索引 + 等值查询（行存在）**
- 加锁类型：**仅 Record Lock**。
- 原因：唯一索引精确命中单行，无歧义，无需锁定间隙。
- 示例：`WHERE id = 10`（`id` 为主键且行存在）→ 仅锁 `id=10` 这条记录。

**场景二：唯一索引 + 等值查询（行不存在）**
- 加锁类型：**Gap Lock**（锁住目标值落入的索引间隙）。
- 原因：行不存在，需防止其他事务在该位置插入，触发幻读。
- 示例：表中存在 `id=5, 20`，查询 `WHERE id = 10`（行不存在）→ 锁住间隙 `(5, 20)`。

**场景三：非唯一索引 + 等值查询**
- 加锁类型：**Next-Key Lock（所有命中行）+ Gap Lock（最后命中行之后的间隙）**。
- 原因：非唯一索引可能有多个命中行，且后续间隙也可能插入同值记录，需全部封堵。
- 示例：索引列 `age` 存在值 `18, 18, 25`，查询 `WHERE age = 18` → 对两个 `age=18` 行加 Next-Key Lock + 对间隙 `(18, 25)` 加 Gap Lock。

**场景四：范围查询 / 无索引全表扫描**
- 加锁类型：**Next-Key Lock（扫描路径上每条记录）+ 边界 Gap Lock**。
- 无索引场景：引擎对全表每行加 Record Lock，等价于表锁，并发极差。
- 示例：`WHERE id > 5 AND id < 20`（表中存在 `10, 15`）→ Next-Key Lock 锁 `(5,10]`、`(10,15]` + Gap Lock 锁 `(15,20)`。

### 设计推论与生产指导

- **必须走索引**：
  - DML 操作不走索引时退化为全表行锁，与表锁等效，严重阻塞并发写。
- **缩短事务**：
  - 行锁在事务提交/回滚时释放，长事务持锁时间长，阻塞后续写操作。
- **规避幻读残留**：
  - 关键范围查询统一使用 `SELECT FOR UPDATE`，明确触发当前读 + Next-Key Lock。

## 死锁排查与解除预案

> [!IMPORTANT]
> **【重难点/生产】多事务环形锁等待，InnoDB 自动检测并主动回滚 Victim**

### 死锁形成与自动检测

- **形成条件**：两个或多个事务形成**循环锁等待**——等待图（Wait-For Graph）出现有向环。
  - Wait-For Graph：节点为活跃事务，有向边 TxA → TxB 表示 TxA 正在等待 TxB 释放某把锁。
  - 示例：TxA 持有 row(id=1) X 锁等待 row(id=2)；TxB 持有 row(id=2) X 锁等待 row(id=1) → TxA→TxB→TxA 成环。
- **InnoDB 检测机制**：每次加锁前 DFS 遍历等待图，检测到环路则立即触发死锁处理。
- **Victim 选择策略**：
  - 选择 `rollback_rows`（已修改行数）最少的事务作为 Victim 强制回滚。
  - 回滚后该事务收到 `ERROR 1213: Deadlock found`，业务层应捕获并重试。
- **超时降级**：`innodb_lock_wait_timeout`（默认 50s）到期后主动回滚等待方，无需形成环路。

### 排查工具链

- **事后分析**（最近一次死锁详情）：
  ```mysql
  SHOW ENGINE INNODB STATUS;
  -- 查看 LATEST DETECTED DEADLOCK 节，含加锁路径与 Victim 信息
  ```
- **实时锁等待**（MySQL 8.0+）：
  ```mysql
  SELECT * FROM performance_schema.data_lock_waits;
  -- 查看当前哪些事务在等待哪些锁
  ```
- **活跃事务扫描**：
  ```mysql
  SELECT * FROM information_schema.INNODB_TRX ORDER BY trx_started;
  -- 排查长事务 / 未提交事务（潜在死锁制造者）
  ```

### 预防方案

- **固定加锁顺序**：
  - 涉及多表或多行的事务，始终按相同的固定顺序加锁。
  - 原理：若所有事务按全序申请资源，Wait-For Graph 必然为 DAG，不可能成环。
- **缩短事务**：
  - 减小事务粒度，让持锁窗口尽可能短，降低并发冲突概率。
- **降级隔离级别至 RC**：
  - RC 级别无 Gap Lock，减少锁范围，降低非必要锁冲突（以允许不可重复读为代价）。
- **走索引**：
  - DML 走索引才能加行锁；不走索引退化为全表锁，显著放大冲突面积。
- **业务层重试兜底**：
  - 捕获 `ERROR 1213`，实现最大 N 次指数退避重试，确保 Victim 事务最终成功。

# 日志

## Redo Log

> [!NOTE]
> **【核心定位】持久性 (D) 的工程实现——WAL 机制的物理载体**

### 本质与核心特征

- **物理日志**：记录数据页级别的物理变更（`space_id + page_no + offset + new_value`），而非 SQL 逻辑语义。
- **顺序追加写**：写入方式为环形文件组的顺序追加，避免数据页随机 IO，性能极高。
- **崩溃恢复依赖**：重启时从最近的 Checkpoint 开始重放，将已提交事务的变更前滚至磁盘。

### WAL 原则

> Write-Ahead Logging：数据页修改落盘之前，对应的 Redo Log 记录必须先落盘。

- 保证即便数据页尚在 Buffer Pool（脏页），只要 Redo Log 已持久化，宕机后即可恢复。
- 将随机写（数据页 `.ibd`）转化为顺序写（Redo Log），是 InnoDB 写性能的核心优化。

### LSN / Checkpoint

- **文件结构**：由 `innodb_log_files_in_group`（默认 2）个固定大小文件组成环形 Ring Buffer，写满后绕回覆写。
- **LSN（Log Sequence Number）**：全局单调递增的字节偏移量，是三方状态同步的共同刻度尺：

  | 位置 | 记录的 LSN | 含义 |
  |---|---|---|
  | Redo Log 文件 | `write_lsn` | 当前已写入到哪里 |
  | Buffer Pool 脏页 | `page_lsn`（页头字段） | 该页最后一次被修改时对应的 LSN |
  | 磁盘数据页 | `page_lsn`（已刷盘） | 该页上次成功落盘时的 LSN |

  恢复时对比 `redo record LSN` 与数据页 `page_lsn`：小于等于则跳过（已包含），大于则重放。

- **Checkpoint 推进机制**：
  - Checkpoint 记录"已安全刷盘的脏页对应的 LSN 水位"，以前的 Redo Log 空间可被覆写回收。
  - 脏页刷盘速度慢于 Redo Log 写入速度时，Checkpoint 推进受阻，环形 Buffer 写满会触发 Sharp Checkpoint，强制阻塞写操作。

## Binlog

> [!NOTE]
> **【核心定位】Server 层的逻辑归档日志——主从复制与数据恢复的核心载体**

### 本质与核心特征

- **逻辑日志**：记录写操作的逻辑语义（行变更或 SQL），与存储引擎无关，MyISAM/InnoDB 均可产生。
- **追加归档写**：写入独立归档文件组（`mysql-bin.000001` 等），文件写满后滚动新建，不覆写，可无限增长。
- **Server 层产生**：在 Redo Log Prepare 之后、Redo Log Commit 之前写入，是 XA 2PC 的中间裁判节点。

### 三种日志格式

| 格式 | 记录内容 | 优点 | 缺陷 |
|---|---|---|---|
| **Statement** | 原始 SQL 语句 | 日志体积小 | 含非确定性函数（`NOW()`/`UUID()`）时主从可能不一致 |
| **Row**（推荐） | 每行的实际变更前后值 | 主从一致性强，可精确回放 | 日志体积大（全量行数据） |
| **Mixed** | 默认 Statement，遇非确定性函数自动切换 Row | 折中方案 | 复杂场景下仍有不一致风险 |

### Binlog  vs Redo Log

| 维度 | Redo Log | Binlog |
|---|---|---|
| **所在层** | InnoDB 引擎层 | Server 层（引擎无关） |
| **日志类型** | 物理日志（页级字节变更） | 逻辑日志（行变更 / SQL） |
| **写入模式** | 环形覆写（固定大小） | 追加归档（可无限增长） |
| **主要用途** | 崩溃恢复（持久性 D） | 主从复制 / 数据归档 / 基于时间点的恢复 |
| **写入时机** | 事务执行过程中持续写入 | Commit 前集中写入（Flush Stage） |

### 主从复制中的角色

- 主库将 Binlog 写入归档文件后，Binlog Dump 线程将增量推送给从库 IO 线程。
- 从库 IO 线程接收后写入本地 **Relay Log**，SQL 线程从 Relay Log 读取并重放，完成数据同步。
- **Row 格式下**：从库直接重放行变更，无需重新执行 SQL，确保幂等性与一致性。

## Undo Log

> [!NOTE]
> **【核心定位】原子性 (A) 的工程实现 + MVCC 版本链的物理底座**

### 本质

Undo Log 是 InnoDB 在**每次写操作之前**生成的旧行镜像，服务于两个完全独立的用途：

| 用途 | 触发方 | 数据范围 |
|---|---|---|
| **事务回滚** | 当前事务自身 | 仅本事务产生的 Undo Log，逆序执行逆向操作 |
| **MVCC 快照读** | 其他事务的读操作 | 多事务的 Undo Log 经 `roll_pointer` 串联成版本链，供快照读查历史版本 |

InnoDB 按操作类型将 Undo Log 分为两类：

| 类型 | 适用操作 | 生命周期 |
|---|---|---|
| **Insert Undo Log** | INSERT | 事务提交后立即可丢弃（插入无历史版本，MVCC 不需要） |
| **Update Undo Log** | UPDATE / DELETE | 事务提交后移交 Purge 线程，等到无活跃 Read View 需要时才清除 |

### 物理存储结构

- **Rollback Segment（回滚段）**：Undo Log 的最高层物理容器，InnoDB 默认维护 128 个回滚段。
- **Undo Segment（撤销段）**：每个事务在回滚段内分配一个或多个 Undo Segment，存储本事务的所有 Undo Log 记录。
- **持久化**：Undo Log 写入 Undo Tablespace（`.ibu` 文件），与数据页一样经由 Buffer Pool 和 Redo Log 保证可靠性。

### Purge 机制

- **Purge 水位线**：取全局所有活跃事务 Read View 的最小 `min_trx_id` 作为水位。
- **回收规则**：`trx_id < 水位线` 的 Update Undo Log 记录对所有活跃事务均不可见，可安全由 Purge 线程异步清除。
- **长事务的危害**：
  - 长事务持有最老的 Read View，钉死水位线，Undo Log 持续堆积，Undo Tablespace 膨胀。
  - 版本链随之变长，快照读遍历链条代价上升，读性能下降。
- **监控指标**：`SHOW ENGINE INNODB STATUS` 中的 `History list length` 反映待清理 Undo Log 记录数，数值过大（通常 > 10000）需立即排查长事务。

## 2PC与崩溃恢复

> [!IMPORTANT]
> **【重难点/架构】内部 XA 机制，解决 Redo Log 与 Binlog 跨系统的分布式一致性难题**

### 双日志一致性
- **Redo Log**：确保 InnoDB 引擎本地的持久性（D）保障。
- **Binlog**：确保主从同步与数据归档的可靠性。
- **不一致风险**：若无原子性保证，主库与从库之间会产生数据错位，破坏集群数据完整性。

### 2PC

1. **Prepare 阶段**：
   - InnoDB 写入变更至 Redo Log，并将状态标记为 **Prepare**。
   - 此时物理变更已持久化，但事务逻辑上尚未结束。
2. **Commit 阶段（第一步：写 Binlog）**：
   - Server 层将事务变更顺序写入 Binlog 文件并执行 `fsync`。
   - **核心判定点**：Binlog 落盘即视作该事务在逻辑上“已不可逆”。
3. **Commit 阶段（第二步：更新 Redo 状态）**：
   - InnoDB 将 Redo Log 中的状态标记修改为 **Commit**。
   - 事务正式终结，释放行锁等相关资源。

### 宕机恢复判定逻辑

系统重启后，InnoDB 会扫描 Redo Log，对处于 **Prepare** 状态的事务结合 **Binlog** 进行判定：

| 事务状态 | Binlog 记录 (XID) | 处理决策 | 判定原则 |
|---|---|---|---|
| **Prepare** | $\times$ 未记录/不完整 | **Rollback** | 保证主从一致；既然从库无法同步，主库必须回滚 |
| **Prepare** | $\surd$ 记录完整 | **Commit** | 以 Binlog 为准；从库已能获取日志，主库须强行冲正 |

- **逻辑裁判**：Binlog 的完整性是衡量事务是否“真正提交”的最高标准。
- **原子性对齐**：通过 Undo Log 辅助执行上述判定后的回滚动作，确保跨日志源的状态绝对一致。

# 高可用架构

## 主从复制架构体系

### 三线程模型与 Relay Log
- **主库 Binlog Dump 线程**：捕获 Binlog 增量并序列化外发。
- **从库 IO 线程**：连接主库并接收日志，顺序写入 **Relay Log**。
- **从库 SQL 线程**：读取 Relay Log 并在引擎层执行重放（Replay）。
- **异步中继缓冲**：Relay Log 物理隔离接收与回放，解耦读写负载。
- **持久化价值**：IO 线程完成落盘即响应，保障主库宕机不丢已传数据。

### 复制模式深度对照

| 复制模式 | ACK 时机 | 一致性 | 性能 | 特色 |
|---|---|---|---|---|
| **异步复制** | 刷入 Binlog 即返回 | $\times$ 弱 | 极高 | 性能优先，极端情况存在丢数风险 |
| **半同步复制** | 至少一从接收落盘 | $\surd$ 较强 | 中 | 兼顾性能与安全，支持自动退化 |
| **增强半同步** | ACK 在 Commit 前 | $\surd$ 强 | 略低 | 解决“幻影提交”，银行/金融级首选 |
| **MGR (组复制)** | 全局多数派共识 | $\surd$ 强一致 | 受限 | 基于 Paxos，支持集群原生自动选主 |

### 主从延迟深度治理

> [!IMPORTANT]
> **【核心目标】通过多线程架构消除从库 SQL 线程在重放（Apply）Relay Log 时的串行瓶颈。**

#### 回放机制
- **优化对象**：将从库原有的单线程 SQL 回放逻辑，演进为“分发+并发执行”的多线程模型。
- **架构实现**：
  1. **Coordinator 调度器**：负责拉取并解析 Relay Log 事务。
  2. **Worker 线程池**：接收调度器分发的事务并执行具体的引擎层变更。
- **价值分析**：彻底解决 IO 线程写入 Relay Log 极快、但 SQL 线程回放极慢导致的流量积压问题。

#### 逻辑时钟
- **安全依据**：利用主库 **组提交 (Group Commit)** 产生的标识，判定事务是否可在从库安全并行。
- **重放逻辑**：凡在主库同一批次提交的事务，Coordinator 视为无冲突，直接分发至 Worker 并发回放。
- **加速路径**：在重放 Relay Log 时，最大限度复现主库的并发拓扑，缩短同步延迟。

#### Writeset
- **探测逻辑**：对事务修改的行（主键/唯一键）进行哈希运算，构建全局 Writeset 冲突集合。
- **重放逻辑**：根据哈希值判定事务间是否存在数据重叠，无重叠事务即绕过顺序限制，强行并发重放。
- **性能巅峰**：摆脱主库提交时间线约束，提供 Relay Log 重放链路下最高维度的并行度。

## 分库分表实战

### 拆分模式定界

#### 垂直拆分
- **垂直分库**：按业务领域拆分物理库（如用户/订单），分摊连接负载。
- **垂直分表**：物理分离变长列或冷字段，提升单页数据密度及缓存命中率。

#### 水平拆分
- **水平分表**：基于 **Sharding Key** 分散行数据，缓解 B+ 树层级 IO 压力。
- **分库分表**：跨实例分布式存储，实现性能与容量的逻辑无限扩展。

### 全局唯一 ID 体系

#### 雪花算法
- **位段构成**：1位符号 + 41位时间戳 + 10位机器ID + 12位序列号。
- **索引友好**：ID 单调趋势递增，与 InnoDB 聚簇索引的页插入形态对齐。

#### 号段模式
- **实现原理**：由中间件/应用预先向数据库申领固定步长的 ID 区间。
- **物理流程**：
  1. 数据库维护 ID 表，保存 `tag`（业务名）与 `max_id`。
  2. 应用通过 `UPDATE ... SET max_id = max_id + step` 原取出号段。
  3. 应用在内存中使用原子计数器分发号段，用完再批量续领。
- **二阶段优化**：引入**双缓冲**机制，在当前号段消耗至阈值时后台预取下一号段。

### 实战疑难深度治理

#### 1. 跨库 Join：应用层聚合
1. 应用层向分片库并发发起独立 SQL 查询，获取两个数据集 A 与 B。
2. 选取数据量较小的集合 B，在内存中构建以关联键为 Key 的 HashMap。
3. 循环遍历集合 A，通过关联键由 HashMap 执行 $O(1)$ 的内存关联。

#### 2. 深分页归并瓶颈

> 游标导航

- **核心思想**：利用 B+ 树叶子节点的双向链表属性，将“随机跳页”改为“顺序游标”。
- **执行逻辑**：
  1. 查询请求必须携带上一页末尾记录的排序字段值（如 `last_id`）。
  2. SQL 构造：`WHERE id > last_id ORDER BY id LIMIT N`。
  3. 性能价值：中间件由位点定位，完全跳过前置的 `OFFSET` 扫描逻辑。

> 延迟关联

- **执行逻辑**：
  1. 仅查询索引列（二级索引），抓取目标 Offset 的 $N$ 个 PK 记录。
  2. 利用获取的 $N$ 个 PK 记录，通过 `IN` 条件回表获取宽行数据。
- **性能价值**：
  - 极大减少了跨分片拉取无效报文的带宽损耗。
  - 避免了在内存中排序归并庞大的宽行结果集。

### 平滑演进方案
1. **镜像构建**：开启增量双写，确保存量同步期间的新增变更在两端对称。
2. **存量对齐**：执行存量快照搬演（如 DataX），补齐双写开始前的历史缺口。
3. **对账校验**：执行全量特征值比对，验证**存量镜像 + 实时增量**的一致性。
4. **灰度切流**：遵循“先读后写”原则分批切入新集，确认平稳后停止老库。
