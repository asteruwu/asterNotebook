# 关键组件

![Kubernetes.drawio](C:\Users\86133\Desktop\学习\dio图表\SRE\Kubernetes.drawio.svg)

- **控制平面组件** - Control Plane Component

  这些组件管理集群的整体状态：

  - **kube-apiserver**：接收指令

  - **etcd**：存储信息，用于所有 API 服务器的数据存储

  - **kube-scheduler**：查找尚未绑定到节点的 Pod，并将每个 Pod 分配给合适的节点

  - **kube-controller-manager**：运行控制器来实现指令任务

  - **cloud-controller-manager**（可选）：管理云资源

- **节点组件** - Node

  在每个节点上运行，维护运行的 Pod 并提供 Kubernetes 运行时环境：

  - **kubelet**：确保Pod及其容器正常运行
  - **Pod**：一组正在运行的容器
  - **k-proxy**（可选）：维护节点上的网络规则以实现 Service 的功能

- **插件** - Addons

  插件扩展了 Kubernetes 的功能。一些重要的例子包括：

  - **Dashboard**（Web界面）：通过 Web 界面进行集群管理
  - **容器资源监控**：用于收集和存储容器指标
  - **集群层面日志**：用于将容器日志保存到中央日志存储

# 核心架构

## 节点

Kubernetes 通过将容器放入在节点（Node）上运行的 Pod 中来执行你的***工作负载***。 节点可以是一个虚拟机或者物理机器，取决于所在的集群配置。 每个节点包含运行Pod所需的服务； 这些节点由***控制面***负责管理。

- **工作负载**：在Kubernetes上运行的应用程序
- **控制面**：控制平面，负责定义、部署容器和管理容器的生命周期

> 节点注册

2. 通过`kubectl`手动添加节点。

## 节点与控制平面之间的通信

- 节点 $$\rightarrow$$ 控制平面

  当 Pod 被实例化时，Kubernetes 自动把公共根证书和一个有效的持有者令牌注入到 Pod 里。 `kubernetes` 服务（位于 `default` 名字空间中）配置了一个虚拟 IP 地址， 用于（通过 `kube-proxy`）转发请求到 API 服务器的 HTTPS 末端。

- 控制平面 $$\rightarrow$$​ 节点

  - API服务器 $$\rightarrow$$ kubelet

    通常用于：

    - 获取 Pod 日志。
    - 挂接（通过 kubectl）到运行中的 Pod。
    - 提供 kubelet 的端口转发功能。

  - API服务器 $$\rightarrow$$ 节点、Pod和服务

    从 API 服务器到节点、Pod 或服务的连接默认为纯 HTTP 方式，因此既没有认证，也没有加密。

## 控制器

在 Kubernetes 中，控制器通过监控集群的公共状态，并致力于将当前状态转变为期望的状态。它通过追踪Kubernetes资源对象代表期望状态的`spec`字段来确保其当前状态接近期望状态。

# Kubernetes对象

## Label & Selector

> [!IMPORTANT]
>
> 此处的Service不是业务服务，而是一个**负载均衡器**（负责流量转发），在找到Pod后分发流量。

Label 是附加在 Kubernetes 对象上的 **键值对（key=value）**。表示kubernetes对象的**属性信息**，用来**识别/分组/查询**对象。

**使用场景**

- Service 通过 label 找到属于自己的 Pod；

  ```yaml
  labels:
    app: frontend
  ```

- 滚动更新区分版本；

  ```yaml
  version: v1
  version: v2
  ```

​	Deployment 在自动更新 Pod 时会根据label分版本区分。

- 监控/日志采集分组；

  Prometheus、Fluent-bit 会用 label 自动收集对应 Pod 的日志/metrics；

- 调度亲和性/反亲和性（affinity）

  - **亲和性：**告诉调度器更希望这个 Pod “倾向” 被放到哪些节点上；

    - **节点亲和性** - 节点通过自己的标签暴露属性，调度器根据Pod要求筛选节点

      ```yaml
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
            - matchExpressions:
                - key: disktype
                  operator: In
                  values:
                    - ssd
      ```

      Pod必须被调度到有标签有`disktype=ssd`的节点上。

    - **Pod亲和性** - 让 Pod 尽量和某些 Pod 放在一起（同节点 or 同拓扑）

      ```yaml
      podAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: backend
            topologyKey: "kubernetes.io/hostname"
      ```

      表示要和`app=backend`的Pod放在同一节点运行。

  - **反亲和性：**告诉调度器不希望这个 Pod 被放到某些节点上。

    - **Pod反亲和性** - 尽量避免和某些 Pod 放在一起，用于高可用

      ```yaml
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: backend
            topologyKey: "kubernetes.io/hostname"
      ```

      表示不能和`app=backend`的Pod放在同一节点运行。

  - **关键字段**

    - **required**DuringSchedulingIgnoredDuringExecution - 硬规则
      - 必须满足
      - 调度器必须放在符合条件的节点
      - 不满足 → Pod 卡 pending
    - **preferred**DuringSchedulingIgnoredDuringExecution - 软规则
      - 最好满足
      - 不满足 → 也能调度，只是优先级降低

---

Selector 是 Kubernetes 用来“按条件筛选对象”的机制，选择对象必须依赖 label。

**使用场景**

- Service找Pod；
  - 自动将**流量导向**所有符合 selector 的 Pod；
- 工作负载（Deployment/Job...）管理Pod；
  - 控制器基于Selector和Label来**查找自己管理的Pod**；
- Pod**找**符合Label的**节点**。

## NameSpace

命名空间用来把一个集群里的资源分成多个逻辑空间，每个空间互不冲突，也可以设置独立的权限、配额、策略。

**应用场景**

- **按环境隔离**

  每个环境可以有自己的Deployment和Service，互不冲突。

  - dev
  - test
  - staging
  - production

- **按团队隔离**

  不同团队互不干扰。

  - team A
  - team B

- **资源配额**

  - 可以对 namespace 设置：
    - 最大 CPU 数量
    - 最大内存
    - 最大对象数量（如 Pod 数量）

- **NetworkPolicy隔离**

  - 网络策略默认基于 Namespace：
    - namespace 内通信放行
    - namespace 间通信可限制

# 核心资源对象

## Pod

**Pod** 是可以在 Kubernetes 中创建和管理的、最小的可部署的计算单元。通常是一组（一个或多个）容器，这些容器共享存储、网络、以及怎样运行这些容器的规约。

> 单容器Pod示例

以下Pod由一个运行镜像 `nginx:1.14.2` 的容器组成。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80
```

### 性质

- 通常并不直接创建Pod，而是通过控制器（如Deployment、Job）**间接创建**；
- 每个 Pod 都旨在运行给定应用程序的**单个实例**。如果希望横向扩展应用程序 （例如，运行多个实例以提供更多的资源），则应该使用多个 Pod，每个实例使用一个 Pod；
- Pod 一般作为相对**临时性**的、用后即抛的**一次性**实体。

### Pod模板

控制器通常使用 **Pod 模板（Pod Template）** 来创建 Pod 并管理它们，它会使用负载对象中的 `PodTemplate` 来生成实际的 Pod。 而`PodTemplate` 是用来运行应用时指定的负载资源的目标状态的一部分。

> 通过Job对象创建Pod示例

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: hello
spec:
  template:
    # 这里是 Pod 模板
    spec:
      containers:
      - name: hello
        image: busybox:1.28
        command: ['sh', '-c', 'echo "Hello, Kubernetes!" && sleep 3600']
      restartPolicy: OnFailure
    # 以上为 Pod 模板
```

> [!NOTE]
>
> 修改 Pod 模板或者切换到新的 Pod 模板都不会对已经存在的 Pod 直接起作用。 

### 资源共享和通信

- **存储**

  - 一个 Pod 可以设置一组共享的**存储卷**。 Pod 中的所有容器都可以访问该共享卷，从而允许这些容器共享数据。

- **通信**

  每个 Pod 都在每个地址族中获得一个唯一的 IP 地址。 Pod 中的每个容器共享网络名字空间，包括 IP 地址和网络端口。

  - **Pod 内**的容器可以使用 `localhost` 互相通信。
  - 当 Pod 中的容器与 **Pod 之外**的实体通信时，它们必须协调如何使用共享的网络资源（例如端口）。

### 生命周期

 Pod 遵循预定义的生命周期，起始于 `Pending` 阶段， 如果至少其中有一个主要容器正常启动，则进入 `Running`，之后取决于 Pod 中是否有容器以失败状态结束而进入 `Succeeded` 或者 `Failed` 阶段。

- **故障恢复**

  - 如果 Pod 中的某个容器失败，`kubelet`可能会尝试重启特定的容器；
  - 如果以集群无法恢复的方式失败，`kubelet` 会删除 Pod 并依赖其他组件提供自动修复；
  - 如果Pod被调用到某个节点后该节点失效，Pod 会被视为不健康，最终`kubelet`会删除 Pod。

- **关联卷**

  -  如果 Pod 因为任何原因被删除，甚至某完全相同的替代 Pod 被创建时， 与它关联的卷也会被删除并重建。

- **阶段状态**

  - Pod 的 `status` 字段是一个PodStatus对象，其中包含一个 `phase` 字段。

  - Pod 的阶段（Phase）是 Pod 在其生命周期中所处位置的简单宏观概述。

    以下是Phase可能的值：

    | 取值                | 描述                                                         |
    | :------------------ | :----------------------------------------------------------- |
    | `Pending`（悬决）   | Pod 已被 Kubernetes 系统接受，但**有一个或者多个容器尚未创建亦未运行**。此阶段包括等待 Pod 被调度的时间和通过网络下载镜像的时间。 |
    | `Running`（运行中） | Pod 已经绑定到了某个节点，Pod 中所有的容器都已被创建。**至少有一个容器仍在运行**，或者正处于启动或重启状态。 |
    | `Succeeded`（成功） | Pod 中的所有容器都已成功结束，并且不会再重启。               |
    | `Failed`（失败）    | Pod 中的所有容器都已终止，并且至少有一个容器是因为失败终止。也就是说，容器以非 0 状态退出或者被系统终止，且未被设置为自动重启。 |
    | `Unknown`（未知）   | 因为某些原因无法取得 Pod 的状态。这种情况通常是因为与 Pod 所在主机通信失败。 |



## Deployment

通常用于管理**无状态**应用，比如部署Web服务/API服务。

### **功能**

- 创建和自动**维护Pod副本**，使其数量保持在**期望值**；
  - 通过更新Deployment配置文件中的 PodTemplateSpec来声明Pod的新状态；
- 滚动**升级**和平滑**回滚**；
  - 如果 Deployment 的当前状态不稳定，回滚到较早的Deployment版本。
- 基于 ReplicaSet 实现**版本控制**与**扩缩容**；
  - 新的 ReplicaSet 会被创建，同时 Deployment 会逐渐增加它的规模， 而减少旧的 ReplicaSet 的规模，确保以受控速率替换 Pod；
  - 清理较旧的不再使用的ReplicaSet。

### **示例**

下面是一个 Deployment 示例。其中创建了一个 ReplicaSet，负责启动三个 `nginx` Pod：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80
```

- 创建名为 `nginx-deployment`（由 `.metadata.name` 字段标明）的 Deployment。
- 创建一个 ReplicaSet，它创建三个（由 `.spec.replicas` 字段标明）Pod 副本。
- `.spec.selector` 字段定义所创建的 ReplicaSet 如何查找要管理的 Pod。 在这里，需要与 Pod 模板中定义的标签（`app: nginx`）相匹配。
- `.spec.template`字段包含以下子字段：
  - Pod 被使用 `.metadata.labels` 字段打上 `app: nginx` 标签。
  - Pod 模板规约（即 `.spec` 字段）指示 Pod 运行一个 `nginx` 容器， 该容器运行版本为 1.14.2 的 `nginx` 镜像。
  - 创建一个容器并使用 `.spec.containers[0].name` 字段将其命名为 `nginx`。

### 回滚

1. 使用`kubectl rollout history deployment/nginx-deployment`检查Deployment修订历史，可能的输出如下：

   ```bash
   deployment.apps/nginx-deployment
   REVISION  CHANGE-CAUSE
   1         <none>
   2         <none>
   ```

   `CHANGE-CAUSE` 的内容是从 Deployment 的 `kubernetes.io/change-cause` 注解复制过来的。 复制动作发生在修订版本创建时。可以通过以下方式设置 `CHANGE-CAUSE` 消息：

   - 使用 `kubectl annotate deployment/nginx-deployment kubernetes.io/change-cause="image updated to 1.16.1"` 为 Deployment 添加注解。
   - 手动编辑资源的清单。

2. 使用`--revision=<>`来查看修订历史的详细信息，比如：

   ```bash
   dantalion@Dantalion:~/k8s$ kubectl rollout history deployment/nginx-deployment --revision=2
   deployment.apps/nginx-deployment with revision #2
   Pod Template:
     Labels:       app=nginx
           pod-template-hash=857d9f6c74
     Annotations:  kubernetes.io/change-cause: image updated to 1.16.1
     Containers:
      nginx:
       Image:      nginx:1.16.1
       Port:       80/TCP
       Host Port:  0/TCP
       Environment: ...
   ```

3. 使用`kubectl rollout undo deployment/nginx-deployment`或`kubectl rollout undo deployment/nginx-deployment --to-revision=1`，可以选择回到上一版本或指定具体版本。

### 编写规约

**核心职责**

- 保证 Pod 副本数（通过 ReplicaSet）
- 滚动升级
- 回滚
- 扩缩容
- 自愈

**结构**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-name
spec:
  replicas: 3               # 多少个副本
  selector:                 # 如何选择要管理的 Pod
  template:                 # Pod 模板（定义 Pod 长什么样）
  strategy:                 # 升级策略（滚动/重建）
  revisionHistoryLimit:     # 保留多少历史版本
  minReadySeconds:          # 多少秒之后认为Pod稳定
```

其中：

- `selector`：必须和`template.metadata.labels`完全匹配，否则无法创建。

- `template`：Pod模板

  ```yaml
      metadata:
        labels:
          app: web
      spec:
        containers:
        - name: web
          image: nginx:1.25
          ports:
          - containerPort: 80
  ```

  包含了Pod所有相关内容，比如标签、Pod上运行的容器（容器名称及其镜像、容器端口）等等。

- `strategy`

  默认滚动升级，比如：

  ```yaml
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  ```

  - `maxUnavailable`：升级时允许最多多少旧 Pod 不可用

  - `maxSurge`：升级时允许最多额外创建多少新 Pod

  也可以选择重建升级。

  ```yaml
  strategy:
  	type: Recreate
  ```

- `template.spec`：Pod级别常用写法

  - 镜像拉取策略

    ```yaml
    imagePullPolicy: IfNotPresent
    ```

  - 资源限制

    ```yaml
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
    ```

  - **探针**

    - readinessProbe

      向kubelet发HTTP请求，检查Pod是否健康。

      ```yaml
      readinessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 5
        periodSeconds: 10
      ```

    - livenessProbe

      kubelet尝试与容器建立TCP连接，看容器端口是否开放。

      ```yaml
      livenessProbe:
        tcpSocket:
          port: 80
        initialDelaySeconds: 15
        periodSeconds: 20
      ```

### **ReplicaSet**

ReplicaSet 的目的是维护一组在任何时候都处于运行状态的 Pod 副本的稳定集合。 因此，它通常用来保证给定数量的、完全相同的 Pod 的可用性。

由于Deployment具有更高的级别，所以一般不单独创建ReplicaSet，而是通过Deployment来管理，在它的spec部分定义应用。

## StatefulSet

StatefulSet 运行一组 Pod，并为每个 Pod 保留一个**稳定的标识**。 这可用于管理**有状态**应用。

###  核心特性

- **有粘性别的PodID**：Pod 基于相同的规约来创建， 但是不能相互替换——无论怎么调度，每个 Pod 都有一个永久不变的 ID。

- **稳定的存储卷**：

  StatefulSet 结合 **VolumeClaimTemplates** 使用 PVC，能够提供稳定、持久的存储：

  - 删除 Pod → 容器销毁
  - 重建 Pod → 挂回原来的 PVC
  - 数据不会丢失

  > [!NOTE]
  >
  > 这也是StatefulSet适用于数据库的原因。

- **有序的、优雅的部署和扩缩**：

  StatefulSet 有严格顺序：

  1. 按顺序创建 Pod：从 0 → 1 → 2
  2. 更新时也是按顺序
  3. 删除时逆序：2 → 1 → 0

  这样能确保每个节点在上线前，前一个节点已经准备好。

- **Headless Service**：提供网络稳定性

  可以保证每个 Pod 有一个固定的 DNS，那么即使Pod被重建或者名字发生变化，对于集群而言的“身份”依然不变。

  > [!IMPORTANT]
  >
  > 对比Deployment中Pod的名称和IP都是随机的，会导致分布式系统和数据库因为 Pod 重建、IP 变化而失联。

  通常搭配

  ```yaml
  serviceName: myapp-headless
  ```

  也就是最开始定义的Service（`clusterIP: None`）。

  它保证每个Pod有一个固定的DNS域名：

  ```
  myapp-0.myapp-headless.default.svc.cluster.local
  myapp-1.myapp-headless.default.svc.cluster.local
  ```

### 组件/编写规约

下面的示例演示了 StatefulSet 的组件。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx #匹配StatefulSet中的.spec.template.metadata.labels
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None
  selector:
    app: nginx
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  selector:
    matchLabels:
      app: nginx # 必须匹配 .spec.template.metadata.labels
  serviceName: "nginx" # 使用定义的Service名称
  replicas: 3 # 默认值是 1
  minReadySeconds: 10 # 默认值是 0
  template:
    metadata:
      labels:
        app: nginx # 必须匹配 .spec.selector.matchLabels
    spec:
      terminationGracePeriodSeconds: 10
      containers:
      - name: nginx
        image: registry.k8s.io/nginx-slim:0.24
        ports:
        - containerPort: 80
          name: web
        volumeMounts:
        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates:
  - metadata:
      name: www
    spec:
      accessModes: [ "ReadWriteOnce" ]
      storageClassName: "my-storage-class"
      resources:
        requests:
          storage: 1Gi
```

- **整体结构**

  两个资源对象：

  1. 名为 `nginx` 的 Headless Service 用来控制网络域名。
  2. 名为 `web` 的 StatefulSet 有一个 Spec，它表明将在独立的 3 个 Pod 副本中启动 nginx 容器。

  它们通过 **label + selector** 和 **serviceName** 绑定在一起，形成**稳定网络身份** + **持久存储**的有状态集群。

---

- **Headless Service**

  - `clusterIP: None`：这使得 Service 不创建负载均衡 IP，而是把 Pod 的 IP 直接暴露给 DNS；

  - `selector: app: nginx`：

    - 只要 Pod 含有 `app=nginx` 的标签，这个 Service 就把它们纳入管理列表；
    - StatefulSet 后面会创建带这个 label 的 Pod；

  - 由于这个Service是Headless的，所以会为每个Pod建立DNS记录，格式为\$(Pod名)-\$(Service名)-\$(命名空间)-svc-\$(集群域名)

    ```
    web-0.nginx.default.svc.cluster.local
    web-1.nginx.default.svc.cluster.local
    web-2.nginx.default.svc.cluster.local
    ```

- **StatefulSet**

  - `selector.matchLabels` + `Pod template.labels`

    这两者必须严格一致，否则 StatefulSet 会报错。

    这是 StatefulSet 管理它所创建 Pod 的身份凭证。

    ```yaml
    matchLabels:
      app: nginx
    
    template.metadata.labels:
      app: nginx
    ```

  - `serviceName: "nginx"`：声明这个StatefulSet**依赖名为 nginx 的 Headless Service 来生成 DNS 名称**。

  - `Pod template`部分：

    - Pod被设置为运行一个名为nginx的容器；
    - `containerPort: 80`对应Service中设置的端口；
    - `volumeMounts`中的`name`与接下来的`volumeClaimTemplates`中的相对应。

  - `volumeClaimTemplates`：StatefulSet 的独特功能，提供稳定存储的关键

    - 每个Pod都有自己独立的PVC，PVC 名字自动带上 Pod 名（Pod名的格式为\$(StatefulSet 名称)-\$(序号)）

      ```
      www-web-0
      www-web-1
      www-web-2
      ```

    - 当 Pod 或者 StatefulSet 被删除时，与 PersistentVolumeClaims 相关联的 PersistentVolume 并不会被删除。

### 部署和扩缩

- 对于包含N个副本的 StatefulSet，当部署 Pod 时，它们是依次创建的，顺序为 {0..N-1}。
- 当删除 Pod 时，它们是逆序终止的，顺序为 {N-1..0}。

- 在将扩缩操作应用到 Pod 之前，它前面的所有 Pod 必须是 Running 和 Ready 状态。
- 在一个 Pod 终止之前，所有的继任者必须完全关闭。

以上文中的配置文件为例，在 nginx 示例被创建后，会按照 web-0、web-1、web-2 的顺序部署三个 Pod：

- 在web-0准备好之前不会部署web-1；
- 如果部署web-2时web-0或web-1运行失败，那么web-2将不会被部署，直到在它之前的Pod都正常运行。

如果要将示例中的StatefulSet更新为`replicas=1`，那么：

- 首先被终止的是web-2；
- 在web-2被完全终止和删除之前，web-1不会被终止；
- 如果web-2已经终止、但是web-0发生运行失败，那么web-1不会终止，直到web-0正常运行。

> 管理策略

可以通过StatefulSet下的`.spec.podManagementPolicy`字段设置。

- **OrderedReady Pod**
  - 默认的管理策略，实现上述提到的部署和扩缩顺序；
- **Parallel**
  - 让 StatefulSet 控制器并行的启动或终止所有的 Pod， 启动或者终止其他 Pod 前，无需等待 Pod 进入 Running 和 Ready 或者完全停止状态。

### 更新

可以通过StatefulSet下的`.spec.updateStrategy`的字段来改变Pod的更新策略。

- **RollingUpdate**
  - 对 StatefulSet 中的 Pod 执行自动的滚动更新。这是默认的更新策略。
  - 按照与 Pod 终止相同的顺序（从最大序号到最小序号）进行，每次更新一个 Pod。
- **OnDelete**
  - 控制器将不会自动更新 StatefulSet 中的 Pod。 用户必须手动删除 Pod 以便让控制器创建新的 Pod，以此来对 StatefulSet 的 `.spec.template` 的变动作出反应。

### 回滚

StatefulSet 使用 ControllerRevision 来维护修订版本历史，从而支持回滚和版本跟踪。

当更新 StatefulSet 的 Pod 模板 (`spec.template`) 时，StatefulSet 控制器：

1. 准备新的 ControllerRevision 对象
2. 存储 Pod 模板和元数据的快照
3. 分配一个递增的修订版本号

> 管理修订版本历史

通过 `.spec.revisionHistoryLimit` 控制保留的修订版本：

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: webapp
spec:
  revisionHistoryLimit: 5  # 保留最近 5 个修订版本
```

- **默认**：如果未指定，保留 10 个修订版本
- **清理**：超过限制时，最早的修订版本会被垃圾回收

> 执行回滚

可以通过以下方式恢复到前一个配置：

```bash
# 查看修订版本历史
kubectl rollout history statefulset/webapp

# 回滚到特定的修订版本
kubectl rollout undo statefulset/webapp --to-revision=3
```

这将会：

- 应用来自修订版本 3 的 Pod 模板
- 使用更新的修订版本号创建新的 ControllerRevision

## DaemonSet

一种特殊的控制器，它的作用是：保证**每个（或指定类型的）Node 上都运行一个 Pod** 实例。

> [!NOTE]
>
> 无论节点新增或删除，DaemonSet 都会自动同步保证 “一机一 Pod”。

### 使用场景

1. **节点级监控**

   在每个节点上运行：

   - **Prometheus Node Exporter**（采集节点指标）

   - **cAdvisor**（容器指标采集）

2. **节点级日志收集**

   收集每个节点上的日志。

3. **网络或存储代理**

   需要在每个节点运行的底层服务：

   - **Calico / Cilium / Weave Net**（CNI 插件）
   - **Kube-proxy**（Service 路由）
   - **Ceph / GlusterFS / OpenEBS Agent**

4. **节点安全与策略**

   例如：

   - **Falco**（安全监控）

   - **Sysdig Agent**

   - **Kubelet metrics exporter**

### 编写规约

以下是一个完整示例：

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata: #定义名称并设置命名空间
  name: node-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: node-exporter #管理含有app: node-exporter标签的Pod
  template:
    metadata:
      labels:
        app: node-exporter #与上文selector.matchLabels字段的内容匹配
    spec:
      containers: #定义在Pod中运行的容器
      - name: node-exporter
        image: prom/node-exporter:v1.7.0
        ports:
        - containerPort: 9100
      tolerations:
      - key: "node-role.kubernetes.io/master" #允许将Pod部署在master节点上
        operator: "Exists"
        effect: "NoSchedule"
      nodeSelector: #选择Linux节点
        kubernetes.io/os: linux
```

- 通过`kind`指定控制器类型；
- `selector`与`.spec.template.metadata.labels`字段的内容必须匹配；
- 通过设置`tolerations`能够让Pod在控制平面节点上运行；
- `nodeSelector`能够限制Pod只在符合条件的节点上运行；
  - 另外地，如果没有指定，那么DaemonSet的Pod将在所有节点上创建Pod。
- `template.spec.containers`定义了要运行的容器。

### 滚动更新

现代版本的 K8s（1.6+）中，DaemonSet 支持 Deployment 风格的滚动更新策略：

```yaml
updateStrategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1
```

- 一次只更新 1 个节点上的 Pod；
- 避免同时停止所有节点的守护进程。

## Job

一种 **一次性任务控制器**，它的作用是：创建并管理一个或多个 **Pod** 来执行特定的任务，直到任务“成功完成”为止。

区别于Deployment、StatefulSet这类长期运行的服务，Job具有以下特点：

- 任务 **执行一次**；
- 执行完毕后 **不会重新启动**；
- 失败可以自动重试；
- 成功的 Pod 会被标记为 **“Completed”**；
- Job 跟踪记录成功完成的 Pod 个数。当数量达到指定的成功个数阈值时，Job结束。

### 编写规约

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: example-job
spec:
  completions: 3
  parallelism: 2
  backoffLimit: 4
  activeDeadlineSeconds: 100
  ttlSecondsAfterFinished: 100
  template:
    spec:
      restartPolicy: Never  # Job任务的Pod必须是 Never 或 OnFailure
      containers:
      - name: worker
        image: busybox
        command: ["sh", "-c", "echo Hello Kubernetes! && sleep 2"]
```

**关键字段**

- `completions`：需要成功执行任务的Pod总数；
- `parallelism`：允许的最大并行Pod数，在需要加速任务完成时可适当增大；
- `backoffLimit`：Pod失败时最多重建次数；
- `activeDeadlineSeconds`：设置一个总执行时间限制，一旦Job运行时间达到这个值，所有Pod都会终止；
  - 能够避免Job卡死。

- `ttlSecondsAfterFinished`：在Job完成的相应时间后，ttl控制器会自动将其清除。
  - 能够减少Job堆积。

## CronJob

用来运行**定时任务**，流程为按照`schedule`触发 $$\rightarrow$$ 创建Job $$\rightarrow$$ Job创建Pod执行任务，**不会直接运行Pod**。

### 编写规约

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
spec:
  schedule: "0 2 * * *"              # 每天凌晨 2 点
  concurrencyPolicy: Forbid          # 上一次没结束就跳过
  startingDeadlineSeconds: 300       # 延迟 5 分钟内还能补跑
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      backoffLimit: 2                # 重试次数
      ttlSecondsAfterFinished: 600   # Job 完成 10 分钟后自动清理
      template:
        spec:
          restartPolicy: Never
          containers:
          - name: backup
            image: alpine
            command: ["sh", "-c", "echo doing backup && exit 0"]
```

**关键字段**

- `schedule`：定时表达式

  ```
  # ┌───────────── 分钟 (0 - 59)
  # │ ┌───────────── 小时 (0 - 23)
  # │ │ ┌───────────── 月的某天 (1 - 31)
  # │ │ │ ┌───────────── 月份 (1 - 12)
  # │ │ │ │ ┌───────────── 周的某天 (0 - 6)（周日到周六）
  # │ │ │ │ │                          或者是 sun，mon，tue，web，thu，fri，sat
  # │ │ │ │ │
  # │ │ │ │ │
  # * * * * *
  ```

  例如 `0 3 * * 1` 表示此任务计划于每周一凌晨 3 点运行。

- `comcurrencyPolicy`：声明了 CronJob 创建的 Job 执行时发生重叠如何处理；
  - `Allow`：默认，允许并发Job执行；
    - 用于大部分实时的日常脚本；
  - `Forbid`：不允许并发执行，如果新Job到了而老Job未完成则忽略新Job的执行；
    - 常用于**数据库备份**等场景，要求任务不重复执行；
  - `Replace`：如果新Job到了则将老Job替换为新的。
- `startingDeadlineSeconds`：设置一个缓冲时间，如果Job没有在规定时间点被调度，则在此缓冲时段内仍可再触发一次；
- `successfulJobsHistoryLimit / failedJobsHistoryLimit`：
  - 控制保留的历史Job数量，减少堆积。

# 服务与网络

## Service

### 原理

Service 为一组动态变化但功能一致的Pod提供稳定的、静态的访问入口。因此使用服务时只需访问静态IP，而无需关心提供服务的具体Pod的变化。

1. **绑定Pod**

   通过配置文件中的标签选择器`selector`字段，找到label与之对应的Pod后将其纳入自己的后端负载池；

2. **生成ClusterIP**

   生成一个虚拟的、稳定的静态IP地址，“代表”与之关联的服务，与提供服务的Pod通过标签**动态绑定**；

3. **kube-proxy维护IP规则**

   持续更新Pod的独立IP地址与Pod实体之间的映射关系，在流量抵达ClusterIP后，提供背后的Pod地址信息；

4. **Linux内核负载均衡**

   根据映射关系以及负载均衡规则分发流量到提供服务的Pod。

### 类型

- **ClusterIP**

  - 集群内部访问用的虚拟 IP。

  - 其他Service类型的基础。

  ```bash
  curl http://my-service:8080
  ```

  集群外不能访问。

  以下是一个典型ClusterIP类型Service的示例：

  ```yaml
  apiVersion: v1
  kind: Pod
  metadata:
    name: appPod
    labels:  # 与下文Service中spec.selector字段内容匹配
      app: myapp
  spec:
    containers:
    - name: app
      image: myapp:latest
      ports: 
        - containerPort: 80
          name: myapp-svc
          
  ---
  apiVersion: v1
  kind: Service
  metadata:
    name: myapp
    namespace: default
    labels:
      app: myapp
  spec:
    type: ClusterIP        # 默认类型，内部服务访问
    selector:
      app: myapp           # 选中所有带 app=myapp 的 Pod
    ports:
    - name: http
      port: 80             # Service 暴露的端口（集群内）
      targetPort: myapp-svc     # 对应容器内端口
      protocol: TCP
  ```

  - 定义了一个基于`myapp`的Pod，将容器端口设置为80，并定义对应的`name`字段；
  - 在Service中引用容器端口`name`字段的内容，可以确保Pod容器监听的端口变化时流量仍然能抵达对应的服务。

- **NodePort**

  Kubernetes 会在 **每个 Node** 上都打开同一个端口，也即**无论用哪个节点的 IP，都能访问到同一个服务。**

  - 每个节点将该端口上的流量代理到Service；
  - 修改目标地址为ClusterIP；到达后根据kube-proxy规则被分配到提供服务的Pod。

- **LoadBalancer**

  在使用支持外部负载均衡器的云平台时，平台会为 Service 提供负载均衡器。

  - 来自外部云平台负载均衡器的流量到达节点；
  - 流量被代理到Service的ClusterIP；
  - 最后被重定向到后端各个 Pod 上。

- **HeadlessService**

  由集群内部的DNS规则为每个Pod生成稳定的域名地址。

  - 不生成ClusterIP，kube-proxy不处理这类服务的流量；
  - 直接通过Pod自己的专属静态域名访问到Pod。

### kube-proxy

由于ClusterIP本质上还是虚拟IP，只是代表了一组动态变化的Pod，仅靠这点**无法使流量真正抵达提供服务的Pod实体**，所以由kube-proxy维护一套**Pod自身IP和Pod实体之间的映射关系**。这套映射关系的更新与维护依赖于kube-proxy

- **监控与发现**
  - kube-proxy 运行在每个节点上，作为一个 DaemonSet 或系统守护进程。
  - 它**持续监听** Kubernetes API Server，关注两类资源的变化：
    - **Service 对象**：当有新的 Service 创建、更新或删除时；
    - **Endpoints/EndpointSlices 对象**：当 Pod 的状态发生变化（创建、销毁、健康状态改变）时；
- **规则生成与应用**
  - 当监控到变化时，kube-proxy 会根据当前模式（iptables、IPVS、userspace）生成相应的网络规则；
  - 将它们同步到Linux系统内核中；
- **负载均衡**（内核负责）
  - Linux内核依赖这套规则转发流量，确保流量转发时会走正确的路由，最终能够抵达提供相应服务的Pod。

### Endpoints

Kubernetes 中的一个 API 对象，它的核心作用是：**记录一个 Service 对应后端所有 Pod 的地址信息**。

- **自动创建与管理**

  创建 Service 时，Kubernetes 会自动创建同名的 Endpoints 对象。

- **存储具体的 IP:Port 列表**

- 根据 Pod 的状态**实时自动更新**

  - Pod 创建 → 添加到 Endpoints

  - Pod 删除 → 从 Endpoints 移除  
  - Pod 不健康 → 从 Endpoints 移除（就绪探针readinessProbe失败）
  - Pod 恢复健康 → 重新添加到 Endpoints

  > [!IMPORTANT]
  >
  > **为什么 readinessProbe 影响 Service 的负载池？**
  >
  > 如果Pod没有准备好，那么它的信息就从Endpoints中被删除，进而kube-proxy更新的路由规则也就不包含以它为终点的，最终内核则无法将流量转发到对应Pod。
  >
  > 这也是为什么**Deployment滚动更新时可以做到服务不中断、流量始终转发正确**的原因。

## Ingress

### 原理

给集群提供 HTTP/HTTPS 层面的入口。根据定义的路由规则，将流量转发给对应的Service。

### Resource

通过yaml文件来定义相关的路由规则，可以使用`kubectl get ingress`来查看。

1. **基于域名路由**

   比如

   ```yaml
   spec:
     rules:
     - host: app.example.com           # 域名
       http:
         paths:
         - path: /                     # 路径
           pathType: Prefix
           backend:
             service:
               name: frontend-service  # 后端 Service
               port:
                 number: 80
   ```

   - 访问`http://app.example.com`路由到$$\rightarrow$$`fronted-service:80`。

2. **基于URL路由**

   比如

   ```yaml
   spec:
     rules:
     - host: api.example.com
       http:
         paths:
         - path: /v1/users
           pathType: Prefix
           backend:
             service:
               name: user-service
               port:
                 number: 8080
         - path: /v1/orders
           pathType: Prefix  
           backend:
             service:
               name: order-service
               port:
                 number: 8080
         - path: /static
           pathType: Prefix
           backend:
             service:
               name: static-service
               port:
                 number: 80
   ```

   - 访问`http://api.example.com/v1/users`路由到$$\rightarrow$$`user-service:8080`；
   - 访问`http://api.example.com/v1/orders`路由到$$\rightarrow$$`order-service:8080`；
   - 访问`http://api.example.com/static`路由到$$\rightarrow$$`static-service:80`。

3. **多域名配置**

   只需对应地多在`spec.rules`中添加一个`host`字段即可。

   ```yaml
   spec:
     rules:
     - host: www.example.com  #域名1
       http:
         paths:
         - path: /
           pathType: Prefix
           backend:
             service:
               name: frontend-service
               port:
                 number: 80
     - host: api.example.com  #域名2
       http:
         paths:
         - path: /
           pathType: Prefix
           backend:
             service:
               name: backend-service
               port:
                 number: 8080
     - host: admin.example.com  #域名3
       http:
         paths:
         - path: /
           pathType: Prefix
           backend:
             service:
               name: admin-service
               port:
                 number: 8081
   ```

4. **HTTPS配置**

   在`spec`中添加`tls`字段，包含域名`hosts`和`secretName`。

   ```yaml
   spec:
     tls:
     - hosts:
       - app.example.com
       secretName: app-tls-secret        # 引用包含证书的 Secret
     rules:
     - host: app.example.com
       http:
         paths:
         - path: /
           pathType: Prefix
           backend:
             service:
               name: frontend-service
               port:
                 number: 80
   ```

### Controller

**Ingress 控制器**是一个实际运行在集群中的**应用程序**，它负责：

- 监听 Ingress Resource的变化
- 解析 Ingress 规则
- 配置底层的负载均衡器/代理服务器
- 处理实际的流量路由

常见有：

- Nginx Ingress Controller（最常用）

  Nginx 是最常用的 Ingress Controller，它是 Kubernetes 中的 **官方推荐实现**之一。它提供了 **高度灵活的负载均衡、SSL 终结、路径重写等功能**。

  ```yaml
  apiVersion: networking.k8s.io/v1
  kind: IngressClass
  metadata:
    name: nginx
  spec:
    controller: k8s.io/ingress-nginx
  ```

- Traefik

  Traefik 是一个 **动态反向代理和负载均衡器**，非常适用于微服务架构。它与 Kubernetes 集成得非常好，提供了更多 **自动化的功能**，例如 **自动发现** 后端服务，并且对 **WebSocket、gRPC** 等协议支持较好。

  ```yaml
  apiVersion: networking.k8s.io/v1
  kind: IngressClass  
  metadata:
    name: traefik
  spec:
    controller: traefik.io/ingress-controller
  ```

---

> 多Ingress控制器场景

比如采用不同的控制器并配置相应的路由规则，在定义Resource时通过`ingressClassName`字段区分：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx-ingress
spec:
  ingressClassName: nginx  # 由 nginx 控制器处理
  rules: [...]
---
apiVersion: networking.k8s.io/v1  
kind: Ingress
metadata:
  name: traefik-ingress
spec:
  ingressClassName: traefik  # 由 traefik 控制器处理
  rules: [...]
```

## NetworkPolicy

**NetworkPolicy** 是 Kubernetes 中用于控制 **Pod 之间通信** 的网络访问控制机制。它可以限制哪些 Pod 可以与其他 Pod 或外部网络进行通信，从而提高 **集群安全性** 和 **流量管理**。

### 原理

- **默认策略**：默认情况下，Kubernetes 中的 Pod 之间可以 **自由通信**，不需要额外的配置。

- **应用 NetworkPolicy 后**：当创建了 NetworkPolicy 时，所有不符合允许规则的流量会被 **拒绝**。NetworkPolicy 只会允许显式指定的流量。

- **网络插件支持**：

  NetworkPolicy 本身只是一个 API 对象，它的具体执行依赖于 CNI 网络插件。

  - **控制平面** - Controller
    - CNI 插件的控制器（如 Calico 的 Typha、Cilium 的 Operator）会 **监听（Watch）** API Server 中 NetworkPolicy 和 Pod 的变化。
    - 控制器根据这些对象的当前状态，计算出最终的网络规则。

  - **数据平面** - Agent
    - 计算出的规则会被下发到每个工作节点上，由该节点上的 **Agent**（如 Calico 的 `felix`、Cilium 的 `cilium-agent`）接收。
    - Agent 将这些抽象的 Kubernetes 规则翻译成节点内核能够理解的具体规则。
      - **最主流的技术是 iptables / nftables**： Agent 会生成一系列复杂的 iptables 规则，这些规则在 `FORWARD` 链（以及可能的 `OUTPUT`、`INPUT` 链）上对数据包进行**过滤**。当数据包进入、离开 Pod 的虚拟网卡（veth pair）时，会经过这些规则链的检查，符合允许规则的数据包被放行，不符合的则被丢弃。

### 配置

一个典型的NetworkPolicy如下：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: test-network-policy
  namespace: default
spec:
  # 1. 选择此策略要应用到的 Pod（策略的目标）
  podSelector:
    matchLabels:
      role: db

  # 2. 策略类型，定义是否管理入站/出站流量
  policyTypes:
  - Ingress
  - Egress

  # 3. 入站规则（白名单）
  ingress:
  - from:
      # 规则来源可以基于以下四种
    - ipBlock:
        cidr: 172.17.0.0/16
        except:
        - 172.17.1.0/24
    - namespaceSelector:
        matchLabels:
          project: myproject
    - podSelector:
        matchLabels:
          role: api
    ports:
    - protocol: TCP
      port: 6379

  # 4. 出站规则（白名单）
  egress:
  - to:
    - ipBlock:
        cidr: 10.0.0.0/24
    ports:
    - protocol: TCP
      port: 5978
```

**关键字段**

- `podSelector`

  - 用于选择应用该 NetworkPolicy 的 Pod。可以使用 **标签选择器**（Label Selector）来指定目标 Pod。
  - 如果没有指定 `podSelector`，则影响 **所有 Pod**。

- `ingress.from`和`egress.to`

  定义了流量的“白名单”。`ingress.from` 指定了“谁”可以访问目标 Pod，`egress.to` 指定了目标 Pod 可以访问“谁”。

  支持三种主要的来源/目标类型，它们可以组合使用（`OR` 关系）：

  - `ipBlock`： 基于 IP 地址段（CIDR）。常用于允许集群外部服务或特定 Node 的访问。
  - `namespaceSelector`： 基于命名空间的标签。选择特定命名空间下的 **所有 Pod** 作为来源或目标。空选择器（`{}`）匹配所有命名空间。
  - `podSelector`： 基于 Pod 的标签。选择与目标 Pod **在同一个命名空间** 下的其他 Pod 作为来源或目标。如果结合 `namespaceSelector`，可以选择其他命名空间中的 Pod。

- `ipBlock`
  - 用于限制源 IP 或目标 IP 地址范围，允许或拒绝来自指定 IP 网段的流量。

### 应用场景

1. **允许特定Pod间通信**

   假设有两个服务：`frontend` 和 `backend`，只允许 `frontend` 服务访问 `backend` 服务，其他服务无法访问 `backend`。

   基于`ingress.from`的`podSelector`实现：

   ```yaml
   spec:
     podSelector:
       matchLabels:
         app: backend    # 只影响 backend 服务
     ingress:
       - from:
           - podSelector:
               matchLabels:
                 app: frontend   # 只允许 frontend 服务访问 backend
   ```

2. **限制出站流量**

   只允许 `frontend` 服务向 `backend` 服务发送流量，禁止所有其他出站流量。

   基于`egress.to`的`podSelector`实现：

   ```yaml
   spec:
     podSelector:
       matchLabels:
         app: frontend  # 只影响 frontend 服务
     egress:
       - to:
           - podSelector:
               matchLabels:
                 app: backend  # 只允许发送流量到 backend 服务
   ```

3. **只允许来自特定 IP 地址的流量**

   只允许来自 IP 范围 `192.168.1.0/24` 的流量访问 `frontend` 服务。

   基于`ingress.from`的`ipBlock`实现：

   ```yaml
   spec:
     podSelector:
       matchLabels:
         app: frontend    # 只影响 frontend 服务
     ingress:
       - from:
           - ipBlock:
               cidr: 192.168.1.0/24   # 只允许来自 192.168.1.0/24 的流量
   ```

4. **限制跨命名空间的流量**

   默认情况下，NetworkPolicy 不会限制不同命名空间之间的通信。如果要限制跨命名空间的流量，需要明确指定源命名空间或目标命名空间。

   基于`ingress.from`或`egress.to`的`namespaceSelector`实现：

   1. `matchLabels`选中单一标准的命名空间：

      ```yaml
      spec:
        podSelector: {}  # 影响所有 Pod
        ingress:
          - from:
              - namespaceSelector:
                  matchLabels:
                    env: production  # 只允许来自 env=production 的命名空间
      ```

   2. `matchExpressions`定义更复杂的选择条件：

      ```yaml
      egress:
          - to:
            - namespaceSelector:
                matchExpressions:
                  - key: namespace
                    operator: In
                    values: ["frontend", "backend"]  # 允许来自namespace=frontend或namespace=backend的命名空间
      ```

5. **禁止所有流量**

   常用于高度安全的隔离环境。

   ```yaml
   spec:
     podSelector: {}  # 影响所有 Pod
     ingress: []      # 不允许任何入站流量
     egress: []       # 不允许任何出站流量
   ```

### 默认行为

1. **默认拒绝一切**： 如果一个 Pod 被任何 NetworkPolicy 通过 `podSelector` 选中，那么对于该策略所管理的流量类型（Ingress/Egress），它将进入“默认拒绝”模式。
   - 例如，一个只定义了 `ingress` 规则的策略应用到 Pod A，那么 Pod A 的 **所有入站流量将被拒绝**（除非在`ingress.from`字段中写入特定的规则）。
2. **隔离是双向的**： NetworkPolicy 的隔离是双向的。即使 Pod A 允许访问 Pod B，如果 Pod B 没有相应的入站规则允许来自 Pod A 的流量，那么通信也会失败。
3. **规则是叠加的**： 多个 NetworkPolicy 可以同时作用于同一个 Pod。结果是这些策略规则的 **并集（OR）**。即，只要流量匹配任何一个策略中的允许规则，它就会被允许。
4. **空选择器的含义**：
   - `spec.podSelector: {}`： 选择当前命名空间下的 **所有 Pod**。
   - `ingress.from - podSelector: {}`： 允许当前命名空间下的 **所有 Pod** 作为来源。
   - `ingress.from - namespaceSelector: {}`： 允许 **所有命名空间** 下的 Pod 作为来源。

## DNS

在 Kubernetes 集群中，Pod 是动态的——它们可能因为故障、滚动更新或扩缩容而被创建或销毁，其 IP 地址也随之改变。通过 IP 地址直接访问服务是不可靠的。因此，Kubernetes 引入了 DNS（域名系统）作为一种服务发现机制，允许通过一个稳定的域名来访问服务，而无需关心后端 Pod 的实际 IP。

### 配置

Kubernetes 默认不自己提供 DNS 服务，而是通过一个**插件**来实现。最核心、最常用的插件是 **CoreDNS**。

- **CoreDNS Pod**： 通常以 `Deployment` 的形式运行在 `kube-system` 命名空间下。
- **CoreDNS Service**： 有一个名为 `kube-dns` 的 Service，其拥有一个稳定的集群 IP（Cluster IP），这个 IP 会在每个 Pod 的 `/etc/resolv.conf` 中被指定为 `nameserver`。

### 解析机制

- **Service的DNS解析**

  - 当你在集群中访问一个 **Service** 时，DNS 解析会将 Service 的名称（如 `web.default.svc.cluster.local`）解析为该 Service 对应的一个或多个 Pod 的 IP 地址。
    1. 基于域名访问**Service**，DNS 查询 `web.default.svc.cluster.local`；
    2. 返回该 **Service** 的虚拟 **ClusterIP**；
    3. **kube-proxy** 将请求路由到实际运行该 Service 的 Pod 上。

- **Headless Service的DNS解析**

  - 如果不想通过ClusterIP来访问服务，而是想直接与后端的Pod通信，则使用 **Headless Service**，也就是不分配虚拟 IP 地址的 Service。

    创建 **Headless Service** 时，需要设置 `clusterIP: None`，这种方式下，DNS 查询会直接返回所有后端 Pod 的 DNS 名称。比如你创建了一个名为`myService`的无头服务，你的Pod名称为`web`，那么可能返回的DNS名称如下：

    ```php
    <Pod名称>.<Service名称>.<namespace>.svc.cluster.local
    web-0.myService.default.svc.cluster.local
    web-1.myService.default.svc.cluster.local
    ```

    这种方式常用于 **StatefulSets** 或者一些需要直接访问 Pod 的场景，比如分布式数据库。

### 记录

> Pod的DNS记录

Kubernetes 集群中的每个 Pod 都有一个唯一的 DNS 名称，可以通过 DNS 查询来访问它。Pod 的 DNS 名称一般遵循以下格式：

```php
<pod-ip>.<namespace>.pod.cluster.local
```

- **pod-ip**：Pod 的 IP 地址。
- **namespace**：Pod 所在的 Kubernetes 命名空间。
- **cluster.local**：这是默认的 DNS 域名（你可以根据集群配置修改它）。

例如，对于一个位于 `default` 命名空间，IP 地址为 172.17.0.3 的 Pod， 如果集群的域名为 `cluster.local`，则 Pod 会对应 DNS 名称：

```php
172-17-0-3.default.pod.cluster.local
```

> Service的DNS记录

```php
<Service名称>.<namespace>.svc.cluster.local
```

- **service-name**：Service 的名称。
- **namespace**：Service 所在的命名空间。
- **svc.cluster.local**：默认的服务 DNS 后缀，`svc` 表示 Service。

### 排障

**常见的 DNS 问题**

- **DNS 解析失败**：检查是否正确配置了 DNS 服务，是否有命名空间或服务名称拼写错误。
- **DNS 延迟或不一致**：可能是 DNS 缓存过期，或者 CoreDNS 负载过高导致响应慢。

**解决方案**

1. 检查CoreDNS是否正常运行

   ```bash
   kubectl get pods -n kube-system -l k8s-app=kube-dns
   ```

2. 通过`kubectl exec - it <pod-name> --sh`进入Pod用系统指令测试解析

   ```bash
   # 安装测试工具 (选一个)
   apk add bind-tools       # Alpine 镜像
   apt update && apt install dnsutils # Debian/Ubuntu 镜像
   
   # 测试解析你的Service
   nslookup <Service名称>
   ```

3. 查看 **CoreDNS** 的日志，了解是否有错误信息

   - 如果是 DNS 服务的配置问题，检查 `coredns` 配置文件，确认插件和解析规则是否正确。

# 存储

**卷** 是一个抽象的存储资源，它提供了一种方法，使得 Pod 中的容器能够**访问持久化的存储空间**。Kubernetes 提供了多种类型的卷，可以让应用程序存储数据，并在容器之间**共享数据**。

## 持久卷

持久卷是为应用提供持久化存储的卷，这些卷的生命周期通常独立于 Pod 的生命周期，可以在 Pod 重启或迁移时保持数据的持久性。持久卷的典型应用场景是数据库或文件存储等需要保留数据的服务。

### PV/PVC/StorageClass

- PV 是集群级别的存储**资源**，代表一个存储卷（如 NFS、Ceph、AWS EBS 等）。

  - PV 持久卷和普通的 Volume 一样， 也是使用卷插件来实现的，只是它们拥有独立于任何使用 PV 的 Pod 的生命周期。

  **回收策略**

  - **`persistentVolumeReclaimPolicy`**

    定义了PV被删除时，如何处理数据。在 PVC 被删除后，PV 会根据其回收策略执行相应的操作。

    1. **`Retain`**：PVC 删除时，PV 不会被删除，数据会保留在 PV 中。
    2. **`Delete`**：PVC 被删除时，PV 会被删除，数据也会随之删除。

- PVC 是用户对持久存储的**声明**，表达用户对存储的请求。

  概念上与 Pod 类似：

  -  Pod 会耗用节点资源，而 PVC 申领会耗用 PV 资源。
  - Pod 可以请求特定数量的资源（CPU 和内存）。同样 PVC 申领也可以请求特定的大小和访问模式。

  **资源访问模式匹配**

  - **`accessModes`**

    定义了 Kubernetes 中一个存储卷的访问权限，指明 Pod 能够以何种方式访问存储卷。

    1. **`ReadWriteOnce`**：只允许单个节点的 Pod 以读写模式访问该卷。适用于单副本应用，如数据库。
    2. **`ReadOnlyMany`**：允许多个节点上的 Pod 以只读方式挂载该卷。适用于共享只读数据的应用场景，如静态网站内容。
    3. **`ReadWriteMany`**：允许多个节点上的 Pod 以读写模式访问该卷。通常需要支持共享存储的后端（如 NFS）。适用于需要在多个 Pod 之间共享和修改数据的场景，例如文件服务器、分布式应用等。
    4. **`ReadWriteOncePod`**：只允许 **单个 Pod** 以读写模式挂载。在集群中的每个节点上，**只能有一个 Pod** 持有对该卷的写权限，确保同一卷不会被多个 Pod 并发写入。

  **存储类匹配**

  - **`storageClassName`**

    StorageClass**为持久化存储添加了一个“质量”或“特性”的维度**，区别于PVC的对于资源量的“大小”的要求，它声明的是对于存储空间“**性能特征**”的要求。比如：

    - `storageClassName: fast-ssd`：存储必须是高性能SSD类型。

> 示例

```yaml
# 1. 定义存储类 (StorageClass)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd  # 存储类的名称，在PVC中【引用】
# 指定供给者，这里使用hostPath的模拟驱动（仅用于演示）
provisioner: kubernetes.io/no-provisioner
# 回收策略：由于是hostPath，我们使用Retain（删除PVC后保留数据）
reclaimPolicy: Retain
# 允许卷扩展（虽然hostPath不支持，但这里展示这个配置项）
allowVolumeExpansion: false
# 指定这个存储类为默认存储类（可选）
# 如果设置为默认，创建PVC时不指定storageClassName则会自动使用这个
# annotations:
#   storageclass.kubernetes.io/is-default-class: "true"

---
# 2. 创建持久卷声明 (PVC) - 用户的"订单"
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-app-pvc  # PVC的名称，在Pod中引用
spec:
  storageClassName: fast-ssd  # 【关键】指定使用哪个存储类
  accessModes:
    - ReadWriteOnce # 声明请求的资源的访问模式
  resources:
    requests:
      storage: 5Gi  # 申请5GB存储（小于等于PV的10GB容量）

---
# 3. 创建使用这个PVC的Pod
apiVersion: v1
kind: Pod
metadata:
  name: my-app-pod
spec:
  containers:
  - name: app-container
    image: nginx:alpine  # 使用一个轻量级的nginx镜像
    ports:
    - containerPort: 80
    volumeMounts:
    - name: app-storage  # 引用下面volumes中定义的卷名称
      mountPath: "/usr/share/nginx/html"  # 将存储挂载到nginx的默认页面目录
  volumes:
  - name: app-storage
    persistentVolumeClaim:
      claimName: my-app-pvc  # 【关键】使用上面创建的PVC
```

> 关键点说明

1. **`storageClassName` 的匹配**：
   - PVC 通过 `storageClassName: fast-ssd` 找到了对应的 StorageClass。
   - PV 通过 `storageClassName: fast-ssd` 声明自己属于这个存储类别。
   - 这样，PVC 就能与符合条件的 PV 进行绑定。
2. **数据持久化**：
   - 删除并重建 Pod，数据仍然保存在 PVC 中，而不是 Pod 内部。

## 临时卷

**临时卷** 是与单个Pod生命周期完全绑定的存储卷。当 Pod 被删除时，临时卷也会被永久销毁，数据无法持久化。

**核心价值**

1. **简单高效**：无需复杂的 PV/PVC 配置
2. **性能优越**：尤其是使用内存作为后端时
3. **资源共享**：完美解决同一 Pod 内容器间数据共享问题
4. **自动清理**：随 Pod 自动清理，无需手动维护

### EmptyDir 

用于存储**同一Pod内多个容器**之间**共享**的临时数据、缓存、日志等，一般存储在 Pod 所在节点的本地磁盘上。

- **基本概念**

  - 在 Pod 被调度到节点时**自动创建的空目录**

  - 初始内容为空

  - 同一 Pod 内的所有容器可以访问卷中的文件

- **配置示例**

  ```yaml
  apiVersion: v1
  kind: Pod
  metadata:
    name: empty-dir-pod
  spec:
    containers:
    - name: writer  # 容器1
      image: alpine
      command: ["sh", "-c", "echo 'Hello' > /cache/data.txt && sleep 3600"]
      volumeMounts:
      - name: cache-volume
        mountPath: /cache
    - name: reader  # 容器2
      image: alpine
      command: ["sh", "-c", "cat /tmp/data.txt && sleep 3600"]
      volumeMounts:
      - name: cache-volume
        mountPath: /tmp
    volumes:  # 定义Pod的临时卷资源，类型为emptyDir
    - name: cache-volume
      emptyDir: {}  # 最简单的空目录卷
  ```

  上述配置文件描述了一个包含两个容器的 **Pod**，它使用了 **emptyDir** 卷。

  - **两个容器共享同一个卷**：`writer` 容器将字符串 `'Hello'` 写入 `/cache/data.txt` 文件，而 `reader` 容器试图读取 `/tmp/data.txt` 文件。由于这两个容器共享同一个 `emptyDir` 卷，它们实际上在共享同一份数据。
  - **容器间共享数据**：通过将同一个卷挂载到多个容器的不同路径（`/cache` 和 `/tmp`），实现容器间的数据共享。即 `writer` 容器写入数据，`reader` 容器读取数据。（注意此处`reader`实际上无法读取到`writer`写入的数据，因为它们的挂载路径不同）

- **高级配置**

  比如将临时卷存储在内存文件系统中。

  ```yaml
  volumes:
  - name: cache-volume
    emptyDir:
      medium: Memory    # 使用内存而不是磁盘
      sizeLimit: 100Mi  # 容量限制（可选）
  ```

### ConfigMap

提供了向 Pod 注入配置数据的方法，能够**将存储在集群中的配置数据挂载到容器的文件系统中**，以便容器在运行时读取。

> 示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-pod
spec:
  containers:
    - name: test
      image: busybox:1.28
      command: ['sh', '-c', 'echo "The app is running!" && tail -f /dev/null']
      volumeMounts:
        - name: config-vol  # 与下文定义的Pod级别的ConfigMap卷名称保持一致
          mountPath: /etc/config
  volumes:
    - name: config-vol
      configMap:  # 需要事先创建/定义
        name: log-config
        items:
          - key: log_level
            path: log_level.conf
```

`log-config` ConfigMap 以卷的形式挂载，并且存储在 `log_level` 条目中的所有内容都被挂载到 Pod 的 `/etc/config/log_level.conf` 路径下。**这个路径来源于卷的 `mountPath` 和 `log_level` 键对应的 `path`。**

### Secret

Secret 数据卷允许将 Secret 对象中的敏感数据（如密码、令牌、密钥）以**文件形式**挂载到 Pod 的容器中。

> 示例

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-volume-pod
spec:
  containers:
  - name: app
    image: nginx:alpine
    volumeMounts:
    - name: secret-vol
      mountPath: "/etc/secrets"
      readOnly: true  # 🔐 重要：设置为只读
  volumes:
  - name: secret-vol
    secret:  # 需要事先创建/定义
      secretName: app-secret  # 引用已存在的Secret
```

## CSI

CSI 是 Kubernetes 和各种存储系统之间的“统一协议”。
它定义了一套 **通用、标准化的接口**，让任何存储系统（云盘、本地盘、NAS、分布式存储等）都可以通过实现 CSI 插件来接入 Kubernetes。

**CSI驱动 $$\Leftrightarrow$$ `storage.provisioner`**

在 StorageClass 中通过`provisioner`字段声明：

```yaml
provisioner: kubernetes.io/aws-ebs
```

kubernetes就知道要使用AWS EBS作存储系统，然后去调用AWS EBS的CSI插件，由插件实现集群与AWS EBS存储系统的交互。

### 常见CSI驱动 | provisioner

| 存储系统   | provisioner 值                 |
| ---------- | ------------------------------ |
| AWS EBS    | `kubernetes.io/aws-ebs`        |
| GCE PD     | `kubernetes.io/gce-pd`         |
| Azure Disk | `kubernetes.io/azure-disk`     |
| NFS        | `kubernetes.io/nfs`            |
| Ceph RBD   | `ceph.com/rbd`                 |
| 本地存储   | `kubernetes.io/no-provisioner` |

以NFS为例：

`nfs` 卷能将 NFS (网络文件系统) 挂载到Pod中。

1. **部署NFS Client Provisioner**

   ```bash
   # 添加 Helm 仓库
   helm repo add nfs-subdir-external-provisioner https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner/
   
   # 安装 Provisioner
   helm install nfs-subdir-external-provisioner \
     nfs-subdir-external-provisioner/nfs-subdir-external-provisioner \
     --set nfs.server=192.168.1.100 \
     --set nfs.path=/nfs/share \
     --set storageClass.name=nfs-dynamic \
     --set storageClass.defaultClass=true
   ```

   注意到，基于指定的NFS的CSI驱动（也即provisioner）：

   - 通过`--set storageClass.name=nfs-dynamic`创建了名为`nfs-dynamic`的存储类；
   - 通过`--set storageClass.defaultClass=true`将默认存储类设置为了`nfs-dynamic`。

   因此下一步不需要创建存储类，NFS provisioner会基于此类再结合PVC中的声明分配资源。

2. **创建Pod，使用动态PVC**

   ```yaml
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata:
     name: nfs-pvc-dynamic
   spec:
     storageClassName: nfs-dynamic  # 使用动态存储类
     accessModes:
       - ReadWriteMany  # 请求的资源允许多节点的读写操作
     resources:
       requests:
         storage: 5Gi  # 请求大小为5g的存储空间资源
   
   ---
   apiVersion: v1
   kind: Pod
   metadata:
     name: nfs-dynamic-pod
   spec:
     containers:
     - name: nginx
       image: nginx:alpine
       volumeMounts:  # 将Pod的卷资源挂载到容器中
       - name: nfs-storage
         mountPath: "/usr/share/nginx/html"
     volumes:  # 定义Pod使用的卷资源
     - name: nfs-storage 
       persistentVolumeClaim:
         claimName: nfs-pvc-dynamic
   ```

   - `PVC`

     - `storageClassName: nfs-dynamic`：指定存储类别

       - 引用名为 `nfs-dynamic` 的 StorageClass

       - 启用**动态供给**模式

       - 决定存储的性能、后端类型等特性

     - `accessModes: ReadWriteMany`：定义访问模式

       - **允许多个节点同时读写**

       - 适合 NFS 这类共享存储

     - `resources.requests.storage: 5Gi`：申请存储容量

   - `Pod`

     - `volumes.name`：定义卷资源的名称

       - **Pod 级别**的卷标识符

     - `persistentVolumeClaim.claimName: nfs-pvc-dynamic`：关联PVC

       - **核心连接点**：将 Pod 与 PVC 关联
       - 必须与已存在的 PVC `name` 一致

     - `containers.volumeMounts.name: nfs-storage`：引用卷名称

       - 必须与 `volumes.name` 匹配

       - 建立容器与卷的连接

     - `containers.volumeMounts.mountPath: "/usr/share/nginx/html"`：容器挂载路径

       - 卷在容器内的具体挂载位置

       - Nginx 默认静态文件目录

### 配置文件结构

#### 持久CSI

- `StorageClass` - 声明使用的CSI驱动

  ```yaml
  apiVersion: storage.k8s.io/v1
  kind: StorageClass
  metadata:
    name: ebs-storage
  provisioner: kubernetes.io/aws-ebs  # 🔥 关键字段！声明后端存储
  parameters:
    type: gp2
  ```

- `PersistentVolume` - 数据卷本身，通常由provisioner生成

- `PersistentVolumeClaim` - 用户声明对数据卷资源的申请

  ```yaml
  apiVersion: v1
  kind: PersistentVolumeClaim
  metadata:
    name: my-data
  spec:
    storageClassName: ebs-storage
    resources:
      requests:
        storage: 10Gi
  ```

#### 临时CSI

卷跟随Pod生命周期，Pod删除则数据卷也消失。

```yaml
spec:
  containers:
    - name: my-frontend
      image: busybox:1.28
      volumeMounts:
      - mountPath: "/data"
        name: my-csi-inline-vol
      command: [ "sleep", "1000000" ]
  volumes:
    - name: my-csi-inline-vol
      csi:
        driver: inline.storage.kubernetes.io
        volumeAttributes:
          foo: bar
```

- `volumeMounts`

  - **`mountPath: "/data"`**
    - 卷在容器内部出现的位置
  - **`name: my-csi-inline-vol`**
    - 指定要挂载进容器的数据卷
    - 名称必须与下面 `volumes:` 中对应

- **`volumes:`**  CSI卷定义

  - **`name: my-csi-inline-vol`**
    - 卷名，与容器里的 `volumeMounts.name` 对应。

  - **`csi.driver: inline.storage.kubernetes.io`**

    - **CSI 插件名称**

    - 决定“真正的存储系统是哪一个”

    - Kubernetes 会调用这个 CSI 驱动来创建/挂载卷

  - **`volumeAttributes.foo: bar`**

    - **CSI 驱动的自定义配置**

    - 内容由该驱动决定

    - Kubernetes **完全不解析，原样转发给 CSI 插件**

# 配置与密钥管理

## ConfigMap

### 概念

ConfigMap 是 Kubernetes 用来存储“非敏感配置数据”的对象。类似于`docker-compose.yml`中为不同服务定义的环境变量`env`。

用来保存：

- **字符串、配置信息、环境变量、命令参数、配置文件内容、JSON/YAML 配置等。**

可以被 Pod 以三种形式使用：

- **环境变量、命令行参数、挂载为文件（Volume）**

> 优势

1. **解耦镜像与配置**

   - 镜像中只存逻辑，不存具体配置

   - 配置变化时不需要重新构建镜像，非常适合 CI/CD

2. **配置集中化、结构清晰**

   - 所有配置都在 K8s 里托管

   - 配置管理一致、可审计、可回滚、可版本化（GitOps）

3. **灵活注入方式**

   - 可作为环境变量、文件挂载、多文件目录、cmd args 注入到容器
   - 对开发者友好，适配各种应用类型

4. **配置动态更新（配合 Deployment 滚动更新）**

   - ConfigMap 更新后，Pod 会自动重启（Hash 变化触发）
   
     - 需要在Pod模板即`spec.template.metadata`下添加字段`annotations`，并注入哈希值：
   
       ```yaml
       metadata:
         annotations:
           configmap-hash: "{{ include (print $.Template.BasePath "/web-config.yaml") . | sha256sum }}"
       ```
   
     - 只要ConfigMap发生变化，那么`configmap-hash`字段内容也会变化；
   
     - 此时Deployment监听到Pod模板更新，于是滚动更新Pod，由此实现加载新配置。
   
   - 也支持热更新（通过**文件挂载**），取决于应用是否支持 reload
   
     - 一般Go、Python、Node.js项目都需要在更新配置文件后重启Pod才可应用新配置；
     - 成熟基础组件如Nginx、Envoy、Prometheus可通过配置在检测到更新时触发自身reload。
   

### 使用场景

- 应用的**环境变量配置**
  - 例如数据库地址、缓存地址、服务端口等。

- 以配置文件的方式**挂载给应用**

  如：

  - Nginx 配置

  - Prometheus 配置

  - 应用的 YAML、JSON 配置文件

  - 自定义规则文件

- 存储应用的**命令行参数**

- 作为多应用**共享的配置中心**
  - 例如网关规则、公共字典、通用配置等。

### 核心结构

1. **key-value形式**

   定义ConfigMap的配置内容时直接书写键值对。

   ```yaml
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: app-config
   data: # 类属性键；每一个键都映射到一个简单的值
     LOG_LEVEL: debug
     PORT: "8080"
   ```

2. **配置文件形式**

   本质还是键值对，只是使用文件名作为"key"，文件内容（具体配置）为"value"。

   ```yaml
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: nginx-conf
   data: # 类文件键
     nginx.conf: |
       server {
         listen 80;
         location / {
           root /usr/share/nginx/html;
         }
       }
   ```

   在这个示例中，就会生成名为`nginx.conf`的配置文件，文件内容为`server{...}`字段。

3. **多个文件形式（目录）**

   可以基于多个键值对（"文件名"："配置内容"）生成多个配置文件。

   ```yaml
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: multi-conf
   data: # 类文件键
     redis.conf: | 
       maxmemory 256mb
     app.json: |
       { "debug": true }
   ```

### 注入

- **整套注入**

  在`containers.envFrom.configMapRef`中直接引用。

  ```yaml
  containers:
  - name: app
    image: myapp
    envFrom:
    - configMapRef:
        name: app-config
  ```

- **按key注入**

  在`containers`字段下定义新的环境变量，通过`configMapRef`引用，并写入要注入的配置的key。

  ```yaml
  env:
  - name: LOG_LEVEL
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: LOG_LEVEL
  ```

- **挂载为文件**

  在`containers.volumeMounts`中声明容器内具体的挂载路径，再基于ConfigMap创建并声明`volumes`字段。

  ```yaml
  volumeMounts:
  - name: config
    mountPath: /etc/myapp/
  volumes:
  - name: config
    configMap:
      name: multi-conf
  ```

  这样容器内就会出现：

  ```nginx
  /etc/myapp/redis.conf
  /etc/myapp/app.json
  ```

> 对比docker-compose和ConfigMap+Deployment+container

- `Dockerfile`构建镜像；`docker compose`把镜像引用、运行配置、容器编排混合在一起编写（服务、`image`、`env`等都集合在一个`docker-compose.yml`中）；

- kubernetes拆成3个模块：
  - **镜像层**
    - 由`containers.image`定义，不关心具体是怎么构建的，只负责引用；
  - **配置层**
    - 由ConfigMap单独存储；
  - **编排层**
    - 使用Deployment/StatefulSet/Service/Ingress等控制器进行统一管理。

## Secret

类似于ConfigMap，但专门用于保存机密数据。相当于**加密配置中心**。

### 特性

1. **安全性**

   - 默认以 **Base64** 存储

   - 支持 **aescbc / kms provider** 强加密，防止node/etcd被盗

   - Pod 内挂载时以 **tmpfs 内存文件系统** 挂载，不会写入节点的磁盘
     - 只要 Pod 删除 / Node 重启：文件消失

   - K8s 会限制 Secret 的访问（RBAC）

     防止应用/用户越权和横向访问：

     - 普通用户不能读取任何 Secret
     - Pod 默认能访问挂载到自身所需的 Secret
     - 一个命名空间的 Secret 不能跨命名空间访问

2. **支持多种身份认证场景**

   - 拉取私有镜像（imagePullSecret）

   - TLS/HTTPS（type: kubernetes.io/tls）

   - ServiceAccount Token（JWT）

   - SSH Key（Opaque）

3. **灵活注入方式**

   与ConfigMap相似：

   - 环境变量

   - Volume 文件

   - Pod 拉取镜像

   - Pod 作为 ServiceAccount 运行时的 token

### 类型

| type                               | 用途                   |
| ---------------------------------- | ---------------------- |
| **Opaque**                         | 通用键值对（最常用）   |
| **kubernetes.io/tls**              | TLS 公私钥             |
| **kubernetes.io/dockerconfigjson** | 镜像仓库凭证           |
| **kubernetes.io/basic-auth**       | 用户名密码             |
| **kubernetes.io/ssh-auth**         | SSH Key                |
| **bootstrap.kubernetes.io/token**  | kubeadm 节点加入 token |

### 创建

1. **通过yaml文件创建** - 最常用

   ```yaml
   apiVersion: v1
   kind: Secret
   metadata:
     name: db-secret
   type: Opaque
   stringData:
     username: admin
     password: mypass123
   ```

   - 使用字段`stringData`，kubernetes会自动把它转为base64到`data`字段。

2. **命令行快速创建**

   ```bash
   kubectl create secret generic db-secret \
       --from-literal=username=admin \
       --from-literal=password=mypass123
   ```

3. **通过文件创建**

   ```bash
   kubectl create secret generic tls-secret \
       --from-file=./tls.crt \
       --from-file=./tls.key
   ```

### 注入

与ConfigMap相似。

- **按key注入**

  ```yaml
  env:
  - name: DB_USER
    valueFrom:
      secretKeyRef:
        name: db-secret
        key: username
  ```

  - 在`containers`字段下定义新的环境变量，通过`valueFrom.secretKeyRef`引用整个Secret，再选中特定的key。

- **挂载为文件**

  ```yaml
  volumes:
  - name: secret-vol
    secret:
      secretName: db-secret
  
  volumeMounts:
  - mountPath: /etc/secret
    name: secret-vol
  ```

  - 基于选定的Secret创建Pod级别的数据卷资源；
  - 将Pod的卷挂载进入容器。

  容器内会生成：

  ```bash
  /etc/secret/username
  /etc/secret/password
  ```

  > [!CAUTION]
  >
  > 挂载为文件的话应该控制容器对于secret文件的权限。

  - `volumes.defaultMode`：设置挂载到容器的 Secret 文件的权限

    - 默认是 0644

    - 推荐改为 0400 或 0440 以提升安全性

    ```yaml
    volumes:
      - name: secret-vol
        secret:
          secretName: db-secret
          defaultMode: 0400
    ```

  - `containers.volumeMounts.readOnly`：设置容器对于挂载的读写权限

    ```yaml
    volumeMounts:
    - mountPath: /etc/secret
      name: secret-vol
      readOnly: true
    ```

- **Docker 私有仓库凭证** - 常见

  在Pod级别的`spec`字段添加：

  ```yaml
  imagePullSecrets:
  - name: myregistry
  ```

  secret创建 | 通过命令行

  ```bash

  ```

  *此处如果要使用yaml文件会需要手写json，用命令行会更加方便。*

- **TLS证书** - 常用于Ingress

    tls.crt: ...

## 探针

探针（Probe）是 Kubernetes 用来**检查容器运行状态**的机制。

### livenessProbe

决定**何时重启容器**。 例如，当应用在运行但无法取得进展时，存活探针可以捕获这类死锁。

- 如果失败多次 → Kubelet 会 **重启容器**
- 保证容器不会一直处于“假死状态”

### readinessProbe

决定容器**何时准备好接受流量**。 这种探针在等待应用执行耗时的初始任务时非常有用； 例如：建立网络连接、加载文件和预热缓存。

- 如果失败 → **Pod 不会被加入 Service 端点列表**
- 不会重启容器
- 常用于应用启动缓慢、依赖数据库、缓存预热等场景

### startupProbe

检查容器内的应用**是否已启动**。 启动探针可以用于对慢启动容器进行存活性检测，避免它们在启动运行之前就被 kubelet 杀掉，给予更多启动时间。

- 如果 startupProbe 存在，readiness/liveness 都会被阻止，直到 startupProbe 成功
- 超时/失败 → 重启

### 关键字段

| 字段                  | 层级            | 作用                                |
| --------------------- | --------------- | ----------------------------------- |
| `initialDelaySeconds` | container.probe | 容器启动后等待多久再开始探测        |
| `periodSeconds`       | container.probe | 探测的频率                          |
| `timeoutSeconds`      | container.probe | 单次探测的超时时间                  |
| `failureThreshold`    | container.probe | 连续失败多少次算探测失败            |
| `successThreshold`    | container.probe | 连续成功多少次算恢复（readiness用） |

### 探测方式

- `gRPC`探针

  - **原理：** K8s 直接发送 gRPC 协议的 Health Check 请求
  - **使用场景：**纯 gRPC 微服务

  ```yaml
  livenessProbe:
    grpc:
      port: 5050        # 对应 main.go 里的 listenPort
    initialDelaySeconds: 5  # 容器启动后等5秒再开始探
    periodSeconds: 10       # 每10秒探一次
  ```

- `exec`探针

  - **原理：** K8s 不发网络请求，而是像管理员一样进入容器执行命令
  - **使用场景：**早期的 gRPC 服务（那时候 K8s 还不支持 gRPC 探针），或者需要检查本地文件的场景

  ```yaml
  livenessProbe:
    exec:
      command: 
      - "/bin/grpc_health_probe" # 容器内的小工具路径
      - "-addr=:5050"            # 告诉工具去连本地的5050端口
    initialDelaySeconds: 5
    periodSeconds: 10
  ```

- `tcpSocket`探针

  - **原理：** TCP 三次握手能建立，就算成功
  - **使用场景：**比如 Redis、MySQL，或者还没来得及实现 Health 接口的 gRPC 服务

  ```yaml
  livenessProbe:
    tcpSocket:
      port: 5050        # 只要这个端口是通的，就认为活着
    initialDelaySeconds: 5
    periodSeconds: 10
  ```

- `httpGet`探针

  - **原理：**Kubelet 发送一个 HTTP GET 请求（类似 `curl`）
  - **使用场景：**绝大多数 Web 服务、REST API、Frontend 服务

  ```yaml
  livenessProbe:
    httpGet:
      path: /_healthz   # Frontend代码里通常会定义这个路由
      port: 8080        # 前端服务的HTTP端口
      httpHeaders:      # (可选) 自定义Header
      - name: X-Custom-Header
        value: Awesome
    initialDelaySeconds: 5
    periodSeconds: 10
  ```

---

### 示例

基于一个Deployment，它的Pod模板（即字段`template`）下的`spec`内容如下：

```yaml
    spec:
      containers:
      - name: demo
        image: demo:1.0
        ports:
        - containerPort: 8080

        # 启动探针
        startupProbe:
          httpGet:
            path: /startup
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
          failureThreshold: 30  # 最长 300 秒

        # 存活探针
        livenessProbe:
          httpGet:
            path: /healthz
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10

        # 就绪探针
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
```

*探测方式均为`httpGet`*

## Context/kubeconfig管理

kubeconfig是一个yaml文件，保存了连接 Kubernetes 集群所需的所有信息。它包含以下模块：

- `clusters`
- `users`
- `contexts`

Context 是 kubeconfig 中的一组“使用配置”：指定你当前要用哪个 cluster + user + namespace。

> 示例

假设现在有两个集群可以连接：

- minikube（本地）
- prod（公司生产）

以下是可能的kubeconfig内容：

```yaml
clusters:  # 可连接的集群列表，包括API Server地址
- name: minikube
  cluster:
    server: https://192.168.0.2:8443
- name: prod
  cluster:
    server: https://api.prod.company.com

users:  # 用户认证配置
- name: minikube-user
  user:
    token: xxx

- name: prod-admin
  user:
    client-certificate: ...
    client-key: ...

contexts:  # 包括所有context组合，集群+用户+命名空间
- name: minikube
  context:
    cluster: minikube
    user: minikube-user
    namespace: default

- name: prod
  context:
    cluster: prod
    user: prod-admin
    namespace: app
```
