## 月度工作总结

随手记录一下自认为有价值的东西。

### 技术经验

1. **gRPC resolver + balancer 框架用法**

   1. 旧方案：自己手搓

      哈哈一开始真的自己 diy 了一个小实现出来，但确实要修改的东西非常多，整理了一下设计思路大概就是这样：

      ![DBHA-v2.drawio](../../../dio图表/SRE/DBHA-v2.drawio.svg)

      看起来非常复杂对吧！两个 `channel`、两个 `list` 的状态维护，处理不好可能还会有锁竞争，重连的逻辑也不好写……总之几百行的代码量，改了很多原本的数据结构，后续测试也没什么头绪，而且不知道会不会有各种各样的边界情况没考虑到，然后就出 bug。再加上月末要交付啊！好着急！就开始和我的 agent 讨论有没有别的办法。

   2. 框架优势

      于是就发现了 gRPC 框架自己有一套成熟的地址解析与管理、负载均衡机制。其实一开始动手实现前知道有这个东西，但当时还不知道会怎么获得 receiver 的地址，对 resolver 的解析也不了解，还想着是不是又得自己 diy resolver，感觉好麻烦，所以就作罢了。

      直到后来 mt 告诉我可以先在配置文件里写静态地址，那就很好办了……最简单的默认的就可以！因为可以拿到裸的 `ip:port`，所以就非常直接，都不需要地址解析，直接丢给 gRPC 的 resolver 就好。之后让 agent 帮我做了，只能说用成熟的框架实在是太太太好了。

      1. 代码量

         比自己手搓少超级多！而且几乎不需要动原本已经定义好的数据结构。只是在初始化 receiver 客户端的时候多注册一个 resolver 和 loadBalancer 就好。全都变成了创建 gRPC 连接的参数，非常简洁轻量，只需要把地址解析好喂给 resolver、为 loadBalancer 选好负载均衡算法就行。特别傻瓜式吧！

      2. 对象定义清晰

         我很喜欢的一点就是，和我自己原来的设计方案相比，框架中的**对象与职责分的特别清晰**。

         resolver 就只负责地址解析，解析好了告诉 balancer；balancer 再基于这些地址自己维护 subconn，和这些地址对应的 receiver 建立连接，最后由内部的 picker 来选择每次 rpc 要用哪个 subconn。

         感觉一切都是井然有序的，**resolver 和 loadBalancer 这种对象是由自己的职责抽象出来得到的**。相比我自己散乱的一堆 channel、list，框架做得简直好了太多。

      3. 实现完整

         这个就无需多言了吧！拆箱即用，很多情况都提前帮你考虑到了。就比如重连采用指数退避的形式，我还在思考要怎么实现的时候，框架已经“解决”了。以后有机会一定要去看看具体的实现吧，但至少功能上肯定是被封装好了，非常之方便！其实也有简单地了解了一下，有一说一，感觉和我的小垃圾设计还挺像的，不过细节处理上我肯定差远了！更何况在这种系统下细节才是最重要的！

2. **测试的标准做法**

   上个月就在“测试”了这个月还在测试！不过这种还是不太一样的，压测有它自己的规则和标准，系统测试又会和压测大不一样。

   1. 功能测试和异常测试，场景写清楚

      - 功能测试

        做之前一定要问清楚自己，这个功能所影响的整个链路，是**从哪个端到哪个端？中间涉及到什么节点？每个节点怎么验证他们是正常工作的？**

        就拿我做的这个 gRPC 上报链路做例子。起点是 probe 通过 `Post()` 方法向 gRPC server 也就是 receiver 上报消息，终点则是最终这些数据要真正落到了 sinker 后的数据库中。这么一想问题就会简单很多，整条链路无非就是 `Probe -> Receiver -> DB`。验证的话就更简单了，只要有清晰的可观测性指标（这里关键的就是结构化日志和 metrics），最终能够到 DB 中查询到与 Probe 上报的吻合的数据记录，那就能证明整条链路是通的。

        除此以外我还验证了由单实例的 probe -> 多实例的 receiver 是负载均衡的。这个也很简单，保持服务正常运行一段较长的时间，然后去查每个 receiver 实例的指标，只要收到的消息数量基本一致，那就说明轮询是有效的。

        总结起来，关键的并不是你怎么实现，而是**你知道自己想看到什么**。我想看到 probe 与 receiver 成功通信，那就把服务跑起来然后去看日志，查数据库的记录；我想看到 probe 到 receiver 是没有流量倾斜的，那就跑启动好几个 receiver 实例，然后去查每个 receiver 的指标。

        **脑子里得先有具体的场景，对于功能实现的期望，然后才有执行的动作**。

      - 异常测试

        这个其实也是。得先有预期，然后才去想测试要怎么做。就比如我希望 `Probe -> Receiver -> DB` 这条链路中的任意节点挂了，再恢复以后，整个流程还能正常跑通，而不是一个死了就全都死了还没法打赢复活赛；又比如我希望 receiver 是高可用的，只要不是全挂了那就还能从别的正常实例上报……那就又是基于不同的**具体场景**去定制测试。

        说白了还是场景要明确，验证起来往往反而是更简单的。

   2. 提交代码前都需要测试，否则就是纸上谈兵

      配置渲染的脚本能否对当前模版正常使用、加了 xxx 参数以后就真的会生效吗、其他功能是否符合预期（会不会影响到已经正常在跑的服务？）这些也都是需要考虑的问题。代码提交的时候从来都不只是“我把这个功能写出来了”这么简单。写出来是一回事，实际验证又是一回事。

      可以尝试在每次提 pr 前都问自己这两个问题：

      1. 功能在测试环境测试过了是符合预期的吗？
      2. 会不会影响到别的服务？

      这样被询问代码细节的时候也可以多一点底气！

3. **注册表与接口**

   这个真的是看代码的时候无意中学到的新的小知识！而且感觉是学校不会讲的、ai 安排学习计划的时候也不会专门加进去的。但好像确实是很常用的一种设计！我现在的理解就是，它们都是一种抽象，只不过注册表是关于创建对象，函数在这里是一种具体的“数据结构”；接口定义的则是对象的统一标准行为。它是抽象的，可以说是没有“实体”的，更像是一种约定俗成的契约。

   本质上其实都是一种解耦！你只需要规定好一套范式，然后标准流程直接按照这套范式来，不需要管背后具体的对象是谁。就比如说，我现在有一个标准流程，**点外卖**。

   对应到注册表，就是**你需要先找到一个可以让你的点外卖的软件**，这个软件可以是京东，可以是淘宝，又或者是美团……总之他们都有一个叫做“点外卖”的函数，并且**都囊括在了这个“可以点外卖的软件”的集合**里。如果你想点外卖了，那就只需要在这个集合里选择你想要的那个元素，然后调用它的“点外卖”函数，就可以完成这个标准流程。

   那点完外卖然后呢？你需要**付钱**。这个时候页面就会给你很多的选项，比如说用微信支付，用支付宝支付，银行卡支付，甚至是好友代付。这些的本质都是为了完成一个动作：付钱。商家不需要关心你用的是哪种支付方式，他只关心**有没有完成付钱这个行为**。这就是**接口**。

   很神奇很巧妙吧！虽然不知道我这样理解对不对，但随着我读过的代码越来越多，总有一天会完全理解！

4. **tcp 自定义协议**

   这个也只能算是略微了解了一下，因为 v1 的消息上报（从 agent 到 gm ）用到了自定义的 tcp 协议。不过我并没有细看，因为好多代码看起来好复杂……而且之于主要的故障探测流程好像并不是那么的重要？
   
   我目前的理解就是，它的优势在于更加简单、轻量，自定义 diy 友好；不过相比直接使用成熟框架（如 http / grpc）代码量会更大。



### 工作小技巧

1. **git 的熟练使用**

   这个月因为已经开始有了比较多的代码提交，而且经常涉及到基于 review 的优化、提交 pr 前的测试，所以感觉在 git 的使用上都熟练了不少。也写点小经验吧！

   1. rebase 太好用太干净太美观！

      真的！能 rebase 就 rebase 吧！相当于原封不动地把分支 2 的 commit 挪到了分支 1 而已（？）而 merge 就还会出现奇奇怪怪的 commit 信息！不过除了这个区别我也不知道他俩还有什么差异了><

   2. 灵活创建分支，区分开发与测试

      这个就没什么好说的了吧！现在的开发模式基本上就是已经定下来了，先用 gtm 创建一个后续要用于提交正式 pr 的分支，再基于这个分支创建一个 test 分支，先把代码写到 test 分支并且提交，测试通过后再 rebase 到 pr 分支。这样既可以在服务器上通过拉取 test 分支的代码直接测试，也可以保证 pr 分支的 commit 信息非常干净，只要确保在 test 分支规范提交就好。个人认为非常实用的小技巧！

   3. `git stash` 完全就是神器！

      这个真是用得好频繁吧？！只要本地有未提交代码，不管是从 upstream 同步最新代码，还是切换分支，都得老老实实先 `git stash` 了才能执行下一步。总之能完好保存本地未提交更改这点实在是太好了，更何况它还能多次保存！非常常用非常实用的！必须给到夯！

2. **实现方案到底要怎么选**

   我的评价是真的得多问。拿捏不了就要问。问 ai 也好，问 mt 也好，一定要问！

   1. 手搓还是成熟框架？

      做 receiver 高可用的时候我就这样踩坑！前面也说过了，自己手搓真的浪费了好多时间而且最后效果也不如成熟框架……事实上和 claude 细细讨论了一下才发现用框架也没那么复杂！要是一开始就问清楚、了解清楚也不至于浪费那么多时间，哪怕去问 mt 要怎么做，也比自己闷头捣鼓了那么久还没个结果好。

   2. `context` 还是 mysql `dsn`？

      这个就是真的狠狠涨教训吧！虽然其实我觉得效果上不如用 `context` 。。本质还是没有超时控制导致的，用 `iptables` 给 mysql 模拟网络故障，无奈 tcp 重传用的时间太久了，不做 fail fast 就连日志提示都没有，消息消费到一半就直接卡在那一动不动，排查起来都很不方便。

      而我又只熟悉 go 啊，这些组件说实话真没多少使用经验。第一反应就是做超时控制的话那就用 `context` 吧，还能自定义超时时间，多方便呢。结果兜兜转转改了一大堆，连接口方法都改了（一开始的 `Save()` 是只传递消息体进去的，现在需要多加一个 `context` 参数）总之一改就是联动改了好多地方。

      然后提交代码，折腾好久，mt 就跟我说，直接用 mysql 的 `dsn` 做参数拼接就好了，有自带的 `ReadTimeout` 和 `WriteTimeout`。改动的地方一下子直接从方法定义和各个地方的调用变成了 mysql 初始化这一个点。确实显得更干净一点吧！也是该一开始就做好调研，看有哪几种实现方式，选不出来再去问的。
